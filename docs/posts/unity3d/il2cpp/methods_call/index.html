    翻译代码函数调用

    在这篇文章中，我们将看看il2cpp.exe如何在托管代码中为方法调用生成C++代码。具体来说有六种不同类型的方法调用：

    • 直接调用实例和静态方法
    • 通过编译时委托调用
    • 通过虚方法调用
    • 通过接口方法调用
    • 通过运行时委托调用
    • 通过反射调用

    在每种情况下，我们将关注生成的C++代码正在做什么，特别是这些指令对性能的影响。

    1. Unity设置

    使用Unity 5.0.1p4版本。在Windows上运行编辑器，并为WebGL平台构建。启用开发者模式选项，并将“启用播放器”选项设置为“完整”值。

    使用一个单独的脚本文件来构建，该脚本文件在上一篇文章中进行了修改，以便我们可以看到不同类型的方法调用。该脚本以接口和类定义开始：

    /* 直接调用实例 */
private void CallDirectly() {
    var important = ImportantFactory();
    important.Method(question);
}
/* 直接点用静态函数 */
private void CallStaticMethodDirectly() {
    Important.StaticMethod(question);
}
/* 委托调用 */
private void CallViaDelegate() {
    var important = ImportantFactory();
    ImportantMethodDelegate indirect = important.Method;
    indirect(question);
}
/* 运行时委托 */
private void CallViaRuntimeDelegate() {
    var important = ImportantFactory();
    var runtimeDelegate = Delegate.CreateDelegate(typeof (ImportantMethodDelegate), important, "Method");
    runtimeDelegate.DynamicInvoke(question);
}
/* 接口调用 */
private void CallViaInterface() {
    Interface importantViaInterface = new Important();
    importantViaInterface.MethodOnInterface(question);
}
/* 反射调用 */
private void CallViaReflection() {
    var important = ImportantFactory();
    var methodInfo = typeof(Important).GetMethod("Method");
    methodInfo.Invoke(important, new object[] {question});
}

private static Important ImportantFactory() {
    var important = new Important();
    return important;
}

void Start () {}

    所有这些都定义好了，生成C++代码将位于项目的Temp\StagingArea\Data\il2cppOutput目录中（只要编辑器保持打开状态）。不要忘记在生成的代码上生成Ctags，以帮助导航。

    由于调用静态方法和实例方法之间的差别很小，我们将在本文的其余部分只关注实例方法，但这些信息也适用于静态方法。

    template <typename R, typename T1>
struct VirtFuncInvoker1
{
    typedef R (*Func)(void*, T1, MethodInfo*);

    static inline R Invoke (MethodInfo* method, void* obj, T1 p1)
    {
        VirtualInvokeData data = il2cpp::vm::Runtime::GetVirtualInvokeData (method, obj);
        return ((Func)data.methodInfo->method)(data.target, p1, data.methodInfo);
    }
};

    对libil2cpp GetVirtualDataData的调用在基于托管代码生成的vtable结构中查找一个虚方法，然后调用该方法。

    为什么我们不使用C++11可变参数模板来实现这些VirtFunctionkerN方法？这看起来像是一个需要可变模板的情况，然而il2cpp.exe生成的C++代码必须与一些C++编译器一起工作，有些编译器还不支持所有C++11功能，包括可变参数模板。至少在这种情况下，我们不认为为C++11编译器派生生成的代码值得增加额外的复杂性。

    为什么这是一个虚方法调用呢？

    我们不是在C#代码中调用实例方法吗？

    回想一下，我们通过C#委托调用实例方法。再看看上面生成的代码。我们要调用的实际方法是通过MethodInfo*（方法元数据）参数传入了ImportantMethodDelegate_m5_MethodInfo。如果我们在生成的代码中搜索名为“ImportantMethodDelegate_Invoke_m5”的方法，我们会看到该调用实际上是对ImportantMethodDelegate类型上的托管Invoke方法的调用。这是一个虚方法，所以我们需要进行虚调用。正是这个ImportantMethodDelegate_logke_m5函数实际上执行调用C#代码中Important实例的Method方法。

    通过对C#代码进行看似简单的更改，我们现在已经从对C++自由函数的单个调用变为多个函数调用，再加上一个虚函数表查找。通过委托调用这一个方法比直接调用同一个方法的开销要大得多。哇哦，这么写代码那真是给一个大嘴巴。

    这里的实际方法调用是通过InterfaceFunctionalker::Invoke函数完成的，该函数位于GeneratedInterfaceFunctionalkers.h文件中。与VirtFunctionker1类类似，InterfaceFunctionker1类通过libil2cpp中的il2cpp::vm::Runtime::GetInterfaceFunctionkeData函数在vtable中执行查找。

    为什么在libil2cpp中接口方法调用对比虚方法调用需要使用不同的API？

    注意InterfaceFunctioner1::Invoke调用不仅要传递方法及方法所需参数，还要传递接口对象L_1。存储不同类型的vtable以便在特定的偏移量处写入接口方法。因此，il2cpp.exe需要提供接口以确定要调用的方法。

    通过以上对比，在IL2CPP中调用虚方法和通过接口调用方法实际上是具有相同的开销。

    7. 结论

    虽然这次分析不能替代实际的分析和衡量，但我们可以通过查看生成的C++代码如何用于不同类型的方法调用来了解任何给定方法调用的开销。很明显我们要尽可能避免通过运行时委托和反射进行调用。
