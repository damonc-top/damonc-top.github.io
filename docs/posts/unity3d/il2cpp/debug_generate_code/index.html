    翻译代码生成调试

    在这篇文章中，我们将探索一些调试IL2CPP生成的C++代码更容易的技巧。涉及到如何设置断点、如何查看字符串和用户自定义类型的内容、确定异常发生的位置。

    在调试过程中，考虑到调试从.NET IL代码创建并生成的C++代码，调试它可能会有点不愉快的体验。然而通过调试这些代码，可以深入了解Unity项目的代码如何在实际目标平台设备上执行的（我们将在文章的最后讨论一些调试托管代码）。

    另外，请准备好项目中生成的代码与此篇文章代码的不同。对于每个Unity版本的IL2CPP，官方都在寻找使生成的c++代码更好、更快、更小的办法。

    1. Unity设置

    在这篇文章中，我在mac OSX上使用了Unity 5.0.1p3。我将使用与关于生成代码的帖子中相同的示例项目，但这次我将使用IL2CPP脚本后端为iOS目标构建。正如我在上一篇文章中所做的那样，我将增加选择开发者模式选项进行构建，以便il2cpp.exe将基于IL代码中的名称生成具有类型和方法名称的C++代码。

    在Unity构建生成Xcode项目的后，我可以在Xcode中打开它（我的版本是6.3.1，但任何最新版本都可以），选择我的目标设备（iPad Mini 3，但任何iOS设备都可以）并在Xcode中构建项目。

    2. 设置断点

    在运行之前，首先在HelloWorld类的Start方法的顶部设置一个断点。正如我们在上一篇文章中看到的，在生成的C++代码中，这个方法的名称是HelloWorld_Start_m3。使用Cmd+Shift+O，并开始键入要在Xcode中查找的方法的名称，然后在其中设置断点。

    我们也可以选择“在XCode中创建符号断点”，并将其设置为在此方法处中断。

    现在，当我运行Xcode项目时，我立即看到它在方法开始时中断。

    如果我们知道方法的名称，我们最好在生成的代码中的方法上设置断点，就像这样。我们还可以在Xcode中的某个生成的代码文件中的特定行设置断点。实际上，所有生成的文件都是Xcode项目的一部分。您可以在Classes/Native目录下的Project Navigator中找到它们。

    3. 查看字符串

    在Xcode中有两种方法可以查看IL2CPP字符串的表示。直接查看字符串的内存，或者调用libil2cpp中的一个字符串实用程序将字符串转换为Xcode可以显示的std::string。让我们看看名为_stringLiteral1的字符串的值（剧透警告：其内容是“Hello，IL2CPP！").

    在生成的Ctags代码中（或者在Xcode中使用Cmd+Ctrl+J），跳转到_stringLiteral1的定义，看到它的类型是Il2CppString_14：

    1
2
3
4
5
6
    struct Il2CppString_14
{
  Il2CppDataSegmentString header;
  int32_t length;
  uint16_t chars[15];
};

    事实上，IL2CPP中的所有字符串都是这样表示的。您可以在object-internals.h头文件中找到Il2CppString的定义。这些字符串结构都包括IL2CPP中托管类型的标准头部分（Il2CppObject，通过Il2CppDataSegmentString typedef访问）字段、第二个是四字节长度字段，第三个是两字节长的字符数组。在编译时定义的字符串，如_stringLiteral1，最终会得到一个固定长度的字符数组，而在运行时创建的字符串有一个已分配的数组。字符串中的字符编码为UTF-16。

    如果我们在Xcode的监视窗口中添加_stringLiteral1，选择View Memory of “_stringLiteral1”选项来查看字符串在内存中的布局。

    在内存查看器中，看到：

    字符串第一段长度是16个字节的Il2CppDataSegmentString对象，第二段是四个字节大小的值为0x000E（索引15开始），第三段是字符串的第一个字符0x0048（'H'）。由于每个字符是两个字节宽，但在这个字符串中，所有的字符都只适合一个字节，Xcode将它们显示在右边，每个字符之间都有0x00相隔。尽管如此，字符串的内容仍然清晰可见。这种查看字符串的方法确实有效，但对于更复杂的字符串来说有点困难。

    我们也可以在Xcode的lldb提示符下查看字符串的内容。utils/StringUtils.h头文件为我们提供了libil2cpp中一些字符串实用程序的接口。具体来说，让我们从lldb提示符调用Utf-16 To Utf-8方法。

    1
    static std::string Utf16ToUtf8 (const uint16_t* utf16String);
    将Il2CppString_14的C++结构中的chars成员传递给这个方法，它将返回一个UTF-8编码的std::string。然后，在lldb提示符下，如果我们使用p命令，打印字符串的内容。 
    1
2
    (lldb) p il2cpp::utils::StringUtils::Utf16ToUtf8(_stringLiteral1.chars)
(std::__1::string) $1 = "Hello, IL2CPP!"

    4. 查看用户定义的类型

    还可以查看用户定义类型的内容，在这个项目的简单脚本代码中，我们已经创建了一个名为Important的C#类型，它有一个名为InstanceIdentifier的字段。如果我在脚本中创建Important类型的第二个实例之后设置断点，我可以看到生成的代码已经将InstanceIdentifier设置为值1，正如预期的那样。

    因此，在生成的代码中查看用户定义类型的内容的方式与在Xcode中的C++代码中的方式相同。

    5. 在生成的代码中中断异常

    在通常情况下这些错误表现为托管异常。正如在上一篇文章中所讨论的，IL2CPP使用C++异常来实现托管异常，因此当Xcode中发生托管异常时，我们可以通过几种方式中断。

    引发托管异常时中断的最简单方法是在il2cpp_codegen_raise_exception函数上设置断点，il2cpp.exe在显式引发托管异常的任何地方都使用该函数。

    让项目运行，当Start中的代码抛出InvalidOperationException异常时Xcode将中断。这是一个查看字符串内容非常有用的地方。如果深入研究ex参数的成员，可以看到它有一个_message_2成员，这是一个表示异常消息的字符串。

    打印这个字符串的值，看看问题是什么：

    1
2
    (lldb) p il2cpp::utils::StringUtils::Utf16ToUtf8(&ex->___message_2->___start_char_1)
(std::__1::string) $88 = "Don't panic"

    请注意，这里的字符串具有与上面相同的c++结构，但是生成的字段的名称略有不同。chars字段名为_start_char_1，其类型为uint16_t，而不是uint16_t[]。它仍然是数组的第一个字符，所以我们可以将其地址传递给转换函数。

    但并非所有托管异常都由生成的代码显式触发。在某些情况下，libil2cpp运行时代码将抛出托管异常，并且它不调用il2cpp_codegen_raise_exception来执行此操作。我们如何捕捉这些异常？

    如果我们在Xcode中使用Debug > Breakpoints > Create Exception Breakpoint创建异常断点，然后编辑断点，我们可以选择C++ exceptions并在抛出Il2CppExceptionWrapper类型的异常时中断。由于这个C++类型用于包装所有托管异常，因此它将允许我们捕获所有托管异常。

    让我们通过在我们的脚本中的Start方法的顶部添加以下两行代码来证明这是可行的：

    1
2
    Important boom = null;
Debug.Log(boom.InstanceIdentifier);

    第二行抛出NullReferenceException。如果我们在设置了异常断点的情况下在Xcode中运行这段代码，我们将看到当抛出异常时Xcode确实会中断。但是断点在libil2cpp的代码中，所以我们看到的只是汇编代码。如果我们看一下调用堆栈，我们可以看到我们需要向上移动几帧到NullCheck方法，该方法由il2cpp.exe注入到生成的代码中。

    从那里，我们可以再向上移动一帧，并看到我们的Important类型的实例确实具有NULL值。

    6. 结尾

    在讨论了调试生成代码的一些技巧之后，可以更好地理解如何使用IL2CPP生成的C++代码来跟踪可能的问题。IL2CPP托管代码调试器在哪里？难道我们不应该能够调试通过设备上的IL2CPP脚本运行的托管代码吗？事实上，这是可能的。我们现在有一个内部的、alpha质量的IL2CPP托管代码调试器。它还没有准备好发布，但它已经在我们的路线图上，所以请继续关注。
