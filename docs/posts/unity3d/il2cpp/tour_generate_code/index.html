    翻译il2cpp代码生成

    这是IL2CPP系列的第二篇博客文章。在这篇文章中，我们将研究il2cpp.exe生成的C++代码。沿着的方式，我们将看到托管类型是如何在本机代码中表示的，看看用于支持.NET虚拟机的运行时检查，看看循环是如何生成的等等。

    我们将讨论一些特定于版本的代码，这些代码在Unity的后续版本中肯定会发生变化。但是，概念将保持不变。

    2. 生成的代码概述

    WebGL构建完成后，生成的C++代码可在我的项目目录的Temp\StagingArea\Data\il2cppOutput目录中找到。一旦编辑器关闭，此目录将被删除。只要编辑器是打开的，这个目录将保持不变，所以我们可以检查它。

    l2cpp.exe程序生成了许多文件，即使是对于这个小项目。我看到4625个头文件和89个C++源代码文件。为了处理所有这些代码，我喜欢使用与Exuberant CTags一起工作的文本编辑器。CTags通常会为这段代码快速生成一个标记文件，这使得导航更容易。

    最初，您可以看到许多生成的C++文件不是来自简单的脚本代码，而是标准库（如mscorlib.dll）中代码的转换版本。正如本系列的第一篇文章中提到的，IL2CPP脚本构建与Mono脚本构建使用相同的标准库代码。请注意，我们每次运行il2cpp.exe时都会转换mscorlib.dll和其他标准库程序集中的代码。这似乎是不必要的，因为代码不会更改。

    但是，IL2CPP脚本构建总是使用字节码剥离来减小可执行文件的大小。因此，即使是脚本代码中很小的更改也可能导致标准库代码的许多不同部分被使用或不被使用，这取决于具体情况。因此，我们每次都需要转换mscorlib.dll程序集。我们正在研究更好的方法来进行增量构建，但现在我们还没有任何好的解决方案。

    3. 托管代码如何映射到生成的C++代码

    对于托管代码中的每个类型，il2cpp.exe将为该类型的C++定义生成一个头文件，并为该类型的方法声明生成另一个头文件。

    3.1. 生成的代码分析

    请注意关于il2cpp.exe转换的所有方法的一些有趣的事情：

    • 这些函数不是C++中的成员函数。所有方法都是自由函数，其中第一个参数是“this”指针。对于托管代码中的静态函数，IL2CPP始终为此第一个参数传递NULL值。通过始终使用“this”指针作为第一个参数来声明方法，我们简化了il2cpp.exe中的方法生成代码，并且对于生成的代码我们使其通过其他函数（如委托）调用该函数会更简单。
    • 每个方法都有一个类型为MethodInfo*的参数，其中包括有关该函数的元数据被用做例如虚方法调用之类的操作。Mono构建脚本时使用当前平台特定的跳转机制(trampolines)来传递此元数据。对于IL2CPP，我们决定避免使用相对于平台的跳转机制来帮助提高可移植性。
    trampolines是什么?

    在计算机编程中，trampoline是一种常见的技术，它可以实现在程序执行过程中跳转到一个不同的函数或代码段，并在执行结束后返回原来的位置继续执行。trampoline通常用于实现一些高级特性，比如动态链接库、函数指针、异常处理等。

    • 所有方法都声明为extern“C”，因此il2cpp.exe有时可以欺骗C++编译器，将所有方法视为具有相同类型。
    • 类型以后缀“_t”命名。方法的命名带有“_m”后缀，并在每个方法名称后附加一个唯一的编号来解决名称冲突。如果用户脚本代码中的内容发生任何更改，这些数字也会更改，因此您不能在每次构建时都依赖它们。

    前两点意味着每个方法至少有两个参数：this指针和MethodInfo指针。这些额外的参数是否会导致不必要的开销？虽然它们确实增加了开销，但到目前为止，我们还没有看到任何迹象表明这些额外的参数会导致性能问题。虽然看起来是这样，但业绩上的差异我们是无法衡量的。

    我们可以使用Ctags跳转到这个ToString方法的定义。它位于Bulk_UnityEngine_0.cpp文件中。该方法定义中的代码看起来与Vector3::ToString()方法中的C#代码不太相似。但是，如果使用ILSpy这样的工具来查看Vector3::ToString()方法的代码，您将看到生成的C++代码看起来与IL代码非常相似。

    为什么不单独生成？

    为什么il2cpp.exe不为每个类型的方法定义生成一个单独的C++文件，就像它为方法声明所做的那样？ 这个Bulk_UnityEngine_0.cpp文件非常大，实际上有20481行！我们发现我们使用的C++编译器在处理大量源文件时遇到了问题。编译4000个.cpp文件比编译80个.cpp文件中的相同源代码花费的时间要长得多。因此，il2cpp.exe将类型的方法定义分组处理，并为每组生成一个C++文件。

    现在跳回到方法声明头文件，注意文件顶部附近的这一行：

    1
    #include "codegen/il2cpp-codegen.h"
    il2cpp-codegen.h文件包含生成的代码用来访问libil2cpp运行时服务的接口。稍后我们将讨论生成的代码使用运行时的一些方式。

    第四行对方法体中使用的任何数组或泛型类型的类型元数据进行惰性初始化。因此名称ObjectU5BU5D_t4是类型System.Object[]的名称。这一部分只执行一次，如果类型已经在其他地方初始化，则通常不执行任何操作，因此我们没有发现生成的代码对性能的任何不利影响。

    这段代码是线程安全的吗？

    如果两个线程同时调用Vector3::ToString()会怎么样？实际上，这段代码没有问题，因为libil2cpp运行时中用于类型初始化的所有代码都可以安全地从多个线程调用。il2cpp_codegen_class_from_type函数有可能（甚至很可能）被多次调用，但它所做的实际工作只会在一个线程上发生一次。在初始化完成之前，方法不会继续执行。所以这个方法prologue是线程安全的。

    这三个运行时检查都是由.NET虚拟机提供保证。Mono构建脚本使用特定于平台的信号机制来处理这些相同的运行时检查，而不是注入代码。对于IL2CPP，我们希望更加平台无关，并支持像WebGL这样的平台，其中没有特定于平台的信号机制，因此il2cpp.exe注入了这些检查。

    这些运行时检查会导致性能问题吗？

    在大多数情况下，我们没有看到对性能的任何不利影响，它们提供了.NET虚拟机所需的好处和安全性。但是在一些特定的情况下，我们看到这些检查会导致性能下降，特别是在tight loop中。我们现在正在研究一种方法，允许对托管代码进行注释，以便在il2cpp.exe生成C++代码时删除这些运行时检查。请继续关注。

    tight loop

    在汇编语言中，一个包含少量指令并迭代多次的循环。 这种大量使用I / O或处理资源的循环，无法与运行在操作系统中的其他程序充分共享。

    1
    for (unsigned int i = 0; i < 0xffffffff; ++ i) {}
    这个短语通常用于指定一个迭代次数很多的循环，这会对程序的性能产生严重影响。可以使用大量的CPU周期。

    只要V_2小于3，goto语句就会跳转到IL_00af标签，该标签是循环体的顶部。您可能会猜到il2cpp.exe当前直接从IL生成C++代码，而不使用中间抽象语法树表示。如果你猜到了这一点，你是正确的。您可能还注意到，在上面的运行时检查部分中，一些生成的代码看起来像这样： 
    1
2
    float L_1 = (__this->___x_1);
float L_2 = L_1;
    显然，L_2变量在这里是不必要的。大多数C++编译器都可以优化掉这个额外的赋值。我们目前正在研究使用AST来更好地理解IL代码的可能性，并为涉及局部变量和for循环的情况生成更好的C++代码。

    8. 结尾

    我们刚刚简单的回顾了由IL2CPP为一个非常简单的项目构建脚本生成C++代码。如果您还没有这样做，我鼓励您深入研究项目中生成的代码。在您探索时，请记住生成的C++代码在Unity的未来版本中看起来会有所不同，因为我们一直在努力改进IL2CPP的构建和运行时性能。

    通过将IL代码转换为C++，我们已经能够在可移植性和高性能代码之间获得很好的平衡。我们可以拥有托管代码的许多对开发人员友好的特性，同时仍然可以获得C++编译器为各种平台提供的高质量机器代码的好处。

    在后面的文章中，我们将探索更多生成的代码，包括方法调用、方法实现的共享和调用native library。但是下一次我们将使用Xcode调试一些生成的iOS 64位构建代码。
