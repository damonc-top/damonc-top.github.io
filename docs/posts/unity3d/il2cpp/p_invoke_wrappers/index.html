    翻译p/invoke及wrapper

    在这篇文章中，我们将探索il2cpp.exe如何生成用于托管代码和本机代码之间互操作的包装方法和类型。

    • 了解可直接复制到本机结构中的类型和不可直接复制到本机结构中的类型之间的区别。
    • 了解字符串和数组封送处理。
    • 了解封送处理的成本。

    在C#中正确地声明P/Invoke可能会稍有困难，理解运行时对封送对象所做的操作是一个谜。由于IL2CPP在生成的C++代码中执行大部分封送处理，因此我们可以通过调试看到它的行为，为故障排除和性能分析提供了更好的洞察力。

    Unity文档讨论了原生插件如何与Unity交互。Mono和Microsoft都提供了大量关于P/Invoke的优秀信息。

extern "C" {
    int Increment(int i) {
        return i + 1;
    }

    bool StringsMatch(const char* l, const char* r) {
        return strcmp(l, r) == 0;

    struct Vector {
        float x;
        float y;
        float z;
    };

    float ComputeLength(Vector v) {
        return sqrt(v.x*v.x + v.y*v.y + v.z*v.z);
    }

    void SetX(Vector* v, float value) {
        v->x = value;
    }

    struct Boss {
        char* name;
        int health;
    };

    bool IsBossDead(Boss b) {
        return b.health == 0;
    }

    int SumArrayElements(int* elements, int size) {
        int sum = 0;
        for (int i = 0; i < size; ++i) {
            sum += elements[i];
        }
        return sum;
    }

    int SumBossHealth(Boss* bosses, int size) {
        int sum = 0;
        for (int i = 0; i < size; ++i) {
            sum += bosses[i].health;
        }
        return sum;
    }
}

    2. marshaling封送

    Marshal就是把一个托管结构（类）序列化成一段非托管的内存数据，然后送到另一个进程（.net中Application domain)中供另一个进程中的函数使用。

    既然IL2CPP已经生成C++代码，为什么我们还需要从C#封送到C++代码呢？虽然生成的C++代码是本机代码，但C#中的类型表示在许多情况下与C++不同，因此IL2CPP运行时必须能够从两边来回转换。il2cpp.exe实用程序对类型和方法都执行此操作。

    在托管代码中，所有类型都可以分为可操作类型-blittable和不可操作类型-non-blittable。

    • blittable的类型在托管代码和本机代码中具有通用的表示形式（例如byte、int、float、Double）。
    • non-blittable的类型在托管代码和本机代码中有不同的表示（例如Boolean、String、Object、Array、Class、T[]、ValueType）。

    因此，blittable的类型可以直接传递给本机代码，但non-blittable的类型在传递给本机代码之前需要进行一些转换。这种转换通常涉及新的内存分配。

    为了告诉托管代码编译器给定的方法是在本机代码中实现的，在C#中使用extern关键字。此关键字与DllImport属性允许托管代码运行时查找native method的定义并调用它。il2cpp.exe实用工具为每个外部方法生成C++ wrapper方法。该wrapper会执行一些重要的任务：

    1. 为native method定义了一个typedef，用于通过函数指针调用方法。
    2. 通过名称解析native method，获得指向该方法的函数指针。
    3. 将参数从其托管表示形式转换为其本机表示形式（如有必要）。
    4. 调用native method。
    5. 将方法的返回值从其本机表示形式转换为其托管表示形式（如有必要）。
    6. 将任何out或ref参数从其本机表示形式转换为其托管表示形式（如有必要）。

//In the Bulk_Assembly-CSharp_0.cpp file, search for the string “HelloWorld_Increment_m3”. The wrapper function for the Increment method looks like this:

extern "C" {int32_t DEFAULT_CALL Increment(int32_t);}
extern "C" int32_t HelloWorld_Increment_m3
(
    Object_t *  __this /* static, unused */, 
    int32_t     ___value, 
    const       MethodInfo* method)
{
    typedef int32_t (DEFAULT_CALL *PInvokeFunc) (int32_t);
    static PInvokeFunc _il2cpp_pinvoke_func;
    if (!_il2cpp_pinvoke_func)
    {
        _il2cpp_pinvoke_func = (PInvokeFunc)Increment;
        if (_il2cpp_pinvoke_func == NULL)
        {
            il2cpp_codegen_raise_exception(il2cpp_codegen_get_not_supported_exception("Unable to find method for p/invoke: 'Increment'"));
        }
    }

    int32_t _return_value = _il2cpp_pinvoke_func(___value);

    return _return_value;
}

    首先，注意原生函数签名的typedef，接受单个int32_t并返回一个int32_t

    1
    typedef int32_t (DEFAULT_CALL *PInvokeFunc) (int32_t);

    接下来，wrapper找到正确的函数指针并将其存储在静态变量中：

    1
    _il2cpp_pinvoke_func = (PInvokeFunc)Increment;

    这里的Increment函数实际上来自一个extern语句（在C++代码中）

    1
    extern "C" {int32_t DEFAULT_CALL Increment(int32_t);}

    在iOS平台native methods会静态链接到单个二进制文件中（由DllImport属性中的“__Internal”字符串指示），因此IL2CPP运行时不会查找函数指针。相反，此extern语句会通知链接器在链接时查找适当的函数。在其他平台上，IL2CPP运行时可以使用特定于平台的API方法执行查找（如果需要）以获得此函数指针。

    实际上，在iOS平台托管代码中不正确的P/Invoke签名将在生成的c++代码中显示为链接器错误。虽然运行时不会发生错误。但也要保证所有P/Invoke签名都需要正确，即使它们在运行时不使用。

    最后，通过函数指针调用本机方法，并返回返回值。请注意，参数是按值传递给native function的，因此本机代码中对其值的任何更改在托管代码中都是不可用的，这正是我们所期望的。

    4. Marshaling a non-blittable type

    对于non-blittable类型，比如string，事情会变得复杂许多。回想一下之前的帖子，IL2CPP中的字符串表示为通过UTF-16编码的两字节字符数组，并以4字节长度值为前缀。这个表示与iOS上C中字符串的char 或wchar_t 表示都不匹配，所以我们必须做一些转换。如果我们看一下StringsMatch方法（生成的代码中的HelloWorld_StringsMatch_m4）：

    1
2
3
    DllImport("__Internal")]
[return: MarshalAs(UnmanagedType.U1)]
private extern static bool StringsMatch([MarshalAs(UnmanagedType.LPStr)]string l, [MarshalAs(UnmanagedType.LPStr)]string r);

    下面是转换结果，可以看到每个字符串参数都将被转换为一个char*（由于UnmangedType.LPStr指令）。

    1
    typedef uint8_t (DEFAULT_CALL *PInvokeFunc) (char*, char*);

    对于第一个参数转换看起来像这样：

    1
2
    char* ____l_marshaled = { 0 };
____l_marshaled = il2cpp_codegen_marshal_string(___l);

    分配一个新的适当长度的字符缓冲区，并将字符串的内容复制到新的缓冲区中。当然，在调用native方法之后，我们需要清理那些分配的缓冲区：

    1
2
    il2cpp_codegen_marshal_free(____l_marshaled);
____l_marshaled = NULL;
    因此，封送一个像string这样的non-blittable类型花费的代价可能很高。

    5. Marshaling a user-defined type

    像int和string这样的简单类型还简单一些，但是更复杂的自定义类型呢？

    5.1 Marshaling a blittable user-defined type

    假设我们想要封送上面的Vector结构，它包含三个浮点值。当且仅当其所有字段都是blittable类型时，自定义的类型才是可直接复制的。因此，我们可以调用ComputeLength（生成的代码中的HelloWorld_ComputeLength_m5），而无需转换参数：

    1
2

    typedef float (DEFAULT_CALL *PInvokeFunc) (Vector_t1 );

// I’ve omitted the function pointer code.

float _return_value = _il2cpp_pinvoke_func(___v);
return _return_value;
    观察上面代码注意到参数是通过值传递的，就像初始示例中参数类型为int时一样。

    如果我们想修改Vector的实例并在托管代码中查看这些更改，我们需要通过引用传递它，就像SetX方法（HelloWorld_SetX_m6）一样：

    这里的Vector参数作为指向本机代码的指针传递。生成的代码有一点繁琐，但它基本上是创建一个相同类型的局部变量，将参数的值复制到局部变量，然后用指向该局部变量的指针调用本机方法。在本机函数返回后，局部变量中的值被复制回参数中，然后该值在托管代码中可用。

    一个不可复制的用户定义类型，如上面定义的Boss类型，也可以被封送，但需要更多的工作。此类型的每个字段都必须封送为其本机表示形式。此外，生成的C++代码需要托管类型的表示形式与本机代码中的表示形式相匹配。

    [DllImport("__Internal")]
[return: MarshalAs(UnmanagedType.U1)]
private extern static bool IsBossDead(Boss b);

    extern "C" bool HelloWorld_IsBossDead_m7 (
    Object_t *  __this /* static, unused */, 
    Boss_t2     ___b, 
    const       MethodInfo* method)
{
    typedef uint8_t (DEFAULT_CALL *PInvokeFunc) (Boss_t2_marshaled);

    Boss_t2_marshaled ____b_marshaled = { 0 };
    Boss_t2_marshal(___b, ____b_marshaled);
    uint8_t _return_value = _il2cpp_pinvoke_func(____b_marshaled);
    Boss_t2_marshal_cleanup(____b_marshaled);

    return _return_value;
}
    参数作为类型Boss_t2传递给wrapper函数，这是Boss结构的生成类型。请注意它以不同的类型传递给本机函数：Boss_t2_marshaled。如果我们跳转到这个类型的定义，我们可以看到它与C++静态库代码中Boss结构体的定义相匹配：

    struct Boss_t2_marshaled
{
    char*   ___name_0;
    int32_t ___health_1;
};

    我们再次使用C#中的UnmanagedType.LPStr指令来指示字符串字段应作为char*进行封送。如果您发现自己在调试non-blittable到本机结构的用户定义类型的问题时，在生成的代码中查看this _marshaled结构非常有帮助。如果字段布局与本机端不匹配，则托管代码中的封送处理指令可能不正确。

    Boss_t2_marshal函数是生成的函数用于封送每个字段，Boss_t2_marshal_cleanup释放封送过程中分配的所有内存。

    5.2 Marshaling a non-blittable user-defined type

    最后，我们将探讨如何封送blittable到本机结构中的类型数组和non-blittable到本机结构中的类型的数组。向SumArrayElements方法传递一个整数数组：

    1
2
    DllImport("__Internal")]
private extern static int SumArrayElements(int[] elements, int size);
    想要封送数组，但由于数组的元素类型是int是可blittable，因此封送它的成本非常小： 
    1
2
    int32_t* ____elements_marshaled = { 0 };
____elements_marshaled = il2cpp_codegen_marshal_array<int32_t>((Il2CppCodeGenArray*)___elements);
    il2cpp_codegen_marshal_array函数只是返回一个指向现有托管数组内存的指针，仅此而已！

    但是封送non-blittable到本机结构中的类型的数组的开销要大得多。SumBossHealth方法传递一个Boss实例数组：

    1
2
    [DllImport("__Internal")]
private extern static int SumBossHealth(Boss[] bosses, int size);
    它的wrapper必须分配一个新数组，然后单独封送每个元素：

    在本机方法调用完成后，所有这些分配也会被清理。

    6.结论

    IL2CPP脚本支持与Mono脚本具有相同的封送行为。由于IL2CPP为外部方法和类型生成对应wrapper，因此可以看到托管到本机互操作调用的成本。对于blittable到本机结构的类型，这种成本通常不会太高，但non-blittable到本机结构的类型会很快使互操作变得非常昂贵。在这篇文章中只是触及了封送的表面。请详细研究生成的代码，以了解如何对返回值和输出参数、本机函数指针和托管委托以及用户定义的引用类型进行封送处理。
