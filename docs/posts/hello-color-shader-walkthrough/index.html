Hello-Color 不是“把模型涂成一个颜色”这么简单。它是顶点-片元 Shader 的最小闭环：CPU 选择材质，Unity 绑定属性，顶点阶段给出裁剪空间位置，片元阶段输出候选颜色，固定功能阶段再决定它能否写入帧缓冲。

目标

前几节已经把 GPU 如何画一帧、固定管线如何演进到可编程管线、Built-in/URP/HDRP 的边界、以及三大 Shader 流派讲清了。接下来写第一段真正能跑的 Shader 代码。

这一节只做一件事：写一个 Unity Built-in 风格的顶点片元 Shader，让任何使用它的物体输出一个材质面板可调的纯色。它不采样纹理，不做光照，不接阴影，也不处理透明。正因为它足够小，每一行代码背后的管线契约才能清晰呈现。

最小闭环

这个 Hello-Color Shader 需要完成四个动作：

  1. 在 ShaderLab 里声明材质属性 _Color。
  2. 在顶点阶段把模型空间顶点变换到裁剪空间。
  3. 在片元阶段读取 _Color 并输出到颜色目标。
  4. 让 Unity 在一次 Draw Call 里把 Mesh、Material、Pass 和 GPU 程序绑定到一起。

换句话说，它不是一个“视觉效果示例”，而是一个渲染契约示例。后面所有纹理、法线、光照、透明、描边、溶解和后处理，都可以看作在这个最小闭环上继续加数据、加状态、加 Pass。

Shader "ShaderAllInOne/HelloColor"
{
    Properties
    {
        _Color ("Color", Color) = (1, 0.35, 0.2, 1)
    }

    SubShader
    {
        Tags { "RenderType"="Opaque" "Queue"="Geometry" }

        Pass
        {
            HLSLPROGRAM
            #pragma vertex vert
            #pragma fragment frag

            #include "UnityCG.cginc"

            float4 _Color;

            struct appdata
            {
                float4 vertex : POSITION;
            };

            struct v2f
            {
                float4 pos : SV_POSITION;
            };

            v2f vert(appdata v)
            {
                v2f o;
                o.pos = UnityObjectToClipPos(v.vertex);
                return o;
            }

            float4 frag(v2f i) : SV_Target
            {
                return _Color;
            }
            ENDHLSL
        }
    }

    Fallback Off
}

这段代码是 Built-in Render Pipeline 的教学版本。它使用 UnityCG.cginc 和 UnityObjectToClipPos，语义也按传统顶点片元 Shader 写法展开。URP/HDRP 中仍然有相同的阶段契约，但 include、宏、常量缓冲布局和 Pass 标签不应该照抄，后文会单独划边界。

能跑的最小 Shader 应该先证明两件事：物体能被正确投影到屏幕上，材质数据能被片元阶段读到。没有这两个闭环，后面的任何效果都无从成立。

外壳

Unity 的 .shader 文件不是纯 HLSL 文件。外层的 ShaderLab 负责描述 Unity 如何找到这个 Shader、材质面板暴露哪些属性、当前硬件和管线应选择哪个 SubShader、一次绘制应执行哪个 Pass。真正运行在 GPU 上的顶点函数和片元函数，只是包在 HLSLPROGRAM 里面的一部分。

Shader 名字

1

Shader "ShaderAllInOne/HelloColor"

这一行给 Shader 一个 Unity 内部可寻址的名字。它会出现在材质的 Shader 下拉菜单中，也会被材质资源序列化引用。这里用斜杠只是为了在菜单里分组，不代表文件夹路径。

工程里不要随意改这个名字。文件名改了，材质通常还能继续引用；Shader 名字改了，旧材质可能找不到目标 Shader，Unity 就会显示粉色错误材质。

Properties

1
2
3
4

Properties
{
    _Color ("Color", Color) = (1, 0.35, 0.2, 1)
}

Properties 是 ShaderLab 层的材质属性声明。它解决三个问题：

  • _Color 是代码侧变量名，HLSL 和 C# 也要用这个名字匹配。
  • "Color" 是 Inspector 面板上显示给人的标签。
  • Color 是属性类型，决定材质面板用颜色拾取器编辑它。
  • (1, 0.35, 0.2, 1) 是默认值，新建材质时会使用它。

注意，Properties 本身不等于 GPU 变量。它只是告诉 Unity：这个 Shader 支持一个可序列化、可编辑、可被材质保存的 _Color 属性。真正让片元函数读到它，还需要在 HLSL 里声明同名变量：

1

float4 _Color;

Properties 里的名字、HLSL 里的变量名、C# 里 SetColor 使用的名字必须一致。少一个下划线、大小写不同、材质使用了别的 Shader，都会造成“Shader 没生效”的误判。

SubShader

2
3

SubShader
{
    Tags { "RenderType"="Opaque" "Queue"="Geometry" }
    ...
}

SubShader 是一组可被 Unity 选择的实现。一个 Shader 文件可以写多个 SubShader，用于适配不同硬件、不同渲染路径或不同质量档。Unity 会从上往下找第一个可用的 SubShader。

本文只有一个 SubShader，因为目标是讲最小闭环，不做硬件分级。这里的两个 Tags 很常见：

Tag含义在本文中的作用
RenderType给替换 Shader、深度纹理等系统提供分类信息声明这是不透明渲染类型
Queue决定渲染队列位置放在普通不透明几何队列

这两个 Tag 不会让 Shader 自动受光，也不会决定最终颜色。它们只是告诉 Unity 这个 Pass 处在怎样的渲染分类里。

Pass

1
2
3
4
5
6

Pass
{
    HLSLPROGRAM
    ...
    ENDHLSL
}

Pass 是一次绘制路径。一个复杂材质可能有多个 Pass，例如主光照 Pass、阴影投射 Pass、深度预写 Pass、描边 Pass。Hello-Color 只有一个 Pass，因此一次普通绘制只需要执行一组顶点和片元程序。

Pass 里除了 HLSL 代码，还可以写渲染状态，例如 Cull、ZTest、ZWrite、Blend、ColorMask。本文没有显式写这些状态，Unity 会使用默认不透明绘制状态：背面剔除、开启深度写入、关闭颜色混合。这也是为什么 _Color.a 在这个例子里不会让物体透明。

程序入口

进入 HLSLPROGRAM 之后，代码才开始描述 GPU 可编程阶段。这里要关注的不是语法花样，而是入口绑定和结构体语义。

Pragma

1
2

#pragma vertex vert
#pragma fragment frag

这两行告诉 Unity 编译器：

  • vert 是顶点着色器入口。
  • frag 是片元着色器入口。

函数名本身没有特殊含义。把 vert 改成 VertexMain、把 frag 改成 FragmentMain 也可以，只要 #pragma 同步修改即可。真正重要的是：顶点入口必须输出硬件能继续处理的位置，片元入口必须输出颜色或其他目标。

Built-in 旧教程里经常看到 CGPROGRAM / ENDCG。本文使用 HLSLPROGRAM / ENDHLSL，因为 Unity 里的 Shader 代码本质上已经按 HLSL 路线发展；但在 Built-in 的历史上下文里，很多 include 文件仍然保留 .cginc 后缀。

appdata

1
2
3
4

struct appdata
{
    float4 vertex : POSITION;
};

appdata 描述顶点着色器从 Mesh 顶点流里读取什么。这里读取一个字段：

  • vertex 是 HLSL 变量名。
  • float4 是数据类型。
  • POSITION 是语义，表示这个字段来自 Mesh 顶点位置通道。

如果后面要加 UV，就会写 float2 uv : TEXCOORD0;。如果要加法线，就会写 float3 normal : NORMAL;。顶点结构体不是固定模板，它应该按当前 Shader 需要的数据最小化声明。

v2f

1
2
3
4

struct v2f
{
    float4 pos : SV_POSITION;
};

v2f 通常表示 vertex-to-fragment，也就是从顶点阶段传给片元阶段的数据。这个例子只传一个 pos，语义是 SV_POSITION。

SV_POSITION 是系统值语义。顶点着色器输出它时，硬件会把它当作裁剪空间位置继续做裁剪、透视除法、视口变换和光栅化。如果没有正确输出 SV_POSITION，GPU 不知道这个顶点该落在屏幕哪里，后面的片元阶段也就无从谈起。

需要补一句细节：在顶点输出里，SV_POSITION 表示裁剪空间位置；如果片元输入也声明 SV_POSITION，它通常会变成屏幕空间相关的位置值。本文不在片元阶段读取它，所以只保留最低必要字段。

语义表

把本例出现的语义压成一张表：

语义所在位置谁使用它作用
POSITIONappdata.vertex顶点输入装配从 Mesh 顶点流读取模型空间位置
SV_POSITIONv2f.pos固定功能阶段提供裁剪空间位置，用于裁剪和光栅化
SV_Targetfrag 返回值输出合并阶段写入当前颜色 Render Target

这就是顶点片元 Shader 的最小契约：顶点阶段给位置，片元阶段给颜色，中间的三角形装配、裁剪、光栅化、深度测试和颜色写入交给硬件。

顶点阶段

顶点阶段解决的问题非常具体：模型顶点原本在物体自己的局部坐标里，必须变换到相机可以投影的裁剪空间。Hello-Color 不做顶点动画，也不修改形状，所以顶点函数只做一次标准矩阵变换。

变换函数

1
2
3
4
5
6

v2f vert(appdata v)
{
    v2f o;
    o.pos = UnityObjectToClipPos(v.vertex);
    return o;
}

UnityObjectToClipPos 来自 UnityCG.cginc。它的核心职责是把模型空间位置变换到裁剪空间。直觉上，它等价于把顶点依次乘上模型矩阵、视图矩阵和投影矩阵：

1

Object Space -> World Space -> View Space -> Clip Space

在很多资料里能看到近似写法：

1

mul(UNITY_MATRIX_MVP, v.vertex)

但实际项目里更建议使用 Unity 提供的函数或宏。原因不是它“更高级”，而是它封装了 Unity 对平台差异、矩阵布局、单通道立体渲染等路径的处理。第一章暂时不展开这些细节，后面讲矩阵和顶点输入输出时会拆。

裁剪空间

裁剪空间不是屏幕像素坐标。UnityObjectToClipPos 输出的是一个 float4，其中 w 分量会参与后续透视除法。硬件会继续完成：

  1. 裁剪：丢弃视锥外的三角形或切割穿过边界的三角形。
  2. 透视除法：把 x/y/z 除以 w，得到 NDC。
  3. 视口变换：把 NDC 映射到屏幕区域。
  4. 光栅化：决定三角形覆盖哪些采样点。

顶点 Shader 只负责把位置送到正确的坐标系，不负责逐像素铺开三角形。这个边界需要明确区分。

片元阶段

片元阶段解决的是“当前片元候选颜色是什么”。它不决定这个颜色一定会出现在屏幕上，因为后面还有深度测试、模板测试、混合和颜色写入掩码。

输出颜色

1
2
3
4

float4 frag(v2f i) : SV_Target
{
    return _Color;
}

frag 的返回类型是 float4，对应 RGBA。语义 SV_Target 表示返回值写入当前绑定的颜色目标，通常是相机颜色缓冲或中间 Render Texture 的第 0 个颜色附件。

这个例子没有使用 i，因为片元颜色不依赖 UV、法线、屏幕位置或光照。所有片元都返回同一个 _Color。这也让它成为很好的调试 Shader：如果一个模型使用它仍然显示异常，问题通常在材质绑定、Pass 选择、渲染队列、深度遮挡、相机剔除层或编译错误，而不是复杂光照公式。

不是像素

Unity 和 Direct3D 语境里常说 Pixel Shader，OpenGL/Vulkan 语境里常说 Fragment Shader。严格说，片元是像素候选样本，不等于最终屏幕像素。一个片元可能因为深度测试失败被丢弃，也可能被透明混合改变贡献方式，还可能在 MSAA 下对应多个采样点。

Hello-Color 的片元函数只给出候选颜色：

1

候选颜色 = _Color

它没有绕过后面的固定功能规则。这个认知很重要，否则看到 return float4(1,0,0,1) 就容易误以为它已经“直接写屏幕”。

属性流动

现在回到 _Color。它从哪里来？为什么材质面板改颜色，片元函数里的 float4 _Color 就会变？这条路径比语法更重要。

材质值

Properties 声明让 _Color 出现在材质面板中。创建材质后，Unity 会把当前颜色值序列化在 Material 资源里。运行时如果没有额外覆盖，本次绘制就使用 Material 里的 _Color。

如果通过 C# 修改材质：

1

renderer.material.SetColor("_Color", Color.cyan);

Unity 会修改该 Renderer 使用的材质实例。这个写法直观，但会实例化材质，项目里要注意材质数量和内存管理。

如果想给单个 Renderer 覆盖颜色，同时不复制材质，常用 MaterialPropertyBlock：

1
2
3
4

var block = new MaterialPropertyBlock();
renderer.GetPropertyBlock(block);
block.SetColor("_Color", Color.cyan);
renderer.SetPropertyBlock(block);

无论来源是 Material 资源，还是运行时覆盖，最终都要在绘制前变成 GPU 可读取的数据。

上传绑定

一次绘制前，Unity 会组织这些信息：

  • 当前 Renderer 的 Mesh。
  • 当前材质使用的 Shader。
  • 本次要执行的 SubShader/Pass。
  • Pass 需要的渲染状态。
  • 材质属性、矩阵、相机参数等常量数据。

然后驱动和图形 API 会把这些数据绑定到 GPU 管线。对于 _Color 这种标量或向量属性，它通常会进入 uniform / constant buffer 一类的常量区域。片元 Shader 中的 float4 _Color; 读取的就是当前 Draw Call 绑定好的那份数据。

flowchart LR
  A[Material: _Color] --> B[Unity Draw Setup]
  C[Mesh: POSITION] --> D[Vertex Shader vert]
  B --> E[GPU Constant Data]
  E --> F[Fragment Shader frag]
  D --> G[Clip / Rasterize]
  G --> F
  F --> H[SV_Target]
  H --> I[Color Buffer]

这里有一个工程上很关键的事实：_Color 默认是一次绘制内共享的常量。不是每个顶点一份，也不是每个片元一份。一个物体的所有片元通常读取同一个 _Color。若需要每个实例不同颜色，可以走 GPU Instancing；若需要每个顶点不同颜色，可以加 COLOR 顶点属性；若需要每个像素不同颜色，通常采样纹理或计算程序化图案。

材质属性的本质是“绘制时绑定的一组常量和资源”。Inspector 只是编辑入口，真正决定结果的是本次 Draw Call 绑定到 GPU 的数据。

管线边界

这段 Shader 明确站在 Built-in Render Pipeline 的教学边界内。它讲的是顶点片元阶段契约，而不是 URP/HDRP 的生产模板。

Built-in

Built-in 中，这段代码可以作为最小 Unlit Shader 使用。它依赖：

  • ShaderLab 的传统 SubShader / Pass 结构。
  • UnityCG.cginc 提供的 UnityObjectToClipPos。
  • 直接声明 float4 _Color; 的材质属性读取方式。
  • 默认不透明 Pass 状态。

它不会自动接入 Built-in 的光照、阴影、Lightmap 或 ForwardAdd Pass。要做受光材质，需要继续写光照计算，或使用 Surface Shader，或读 Unity 内置光照 include。那些不是 Hello-Color 的职责。

URP

URP 仍然可以写手工 HLSL 顶点片元 Shader，但不能简单照搬本文代码。常见差异包括：

位置Built-in 示例URP 常见写法
IncludeUnityCG.cgincPackages/com.unity.render-pipelines.universal/ShaderLibrary/Core.hlsl
坐标变换UnityObjectToClipPosTransformObjectToHClip
材质常量直接 float4 _Color;CBUFFER_START(UnityPerMaterial)
Pass 标签可省略 LightMode通常需要明确 LightMode
批处理目标传统路径需要考虑 SRP Batcher 常量布局

URP 的最小 Unlit Shader 也能很短，但它应该按 URP 的库和常量缓冲规范写。否则可能出现能编译但不走预期 Pass、SRP Batcher 失效、Shader Graph/Renderer Feature 体系难以协作等问题。

HDRP

HDRP 的约束更强。HDRP 是高保真 SRP，材质系统、光照模型、Pass 结构、Shader Library 都很重。教学上可以用类似 Hello-Color 的逻辑理解顶点和片元，但生产项目通常不建议从 Built-in 片段手工迁移到 HDRP。

如果项目在 HDRP 中只需要一个纯色材质，优先考虑 HDRP/Lit、HDRP/Unlit 或 Shader Graph。如果要写自定义 Pass 或全屏效果，应该进入 HDRP 的 Custom Pass 和对应 include 体系，而不是复制本文代码。

扩展性说明：从 Hello-Color 长到生产材质

Hello-Color 最值得保留的是边界清晰，而不是代码长度短。

如果要扩展它，建议按数据来源逐步加能力：先加 UV 和纹理采样，再加法线和基础光照，再加透明或裁剪，最后再考虑多 Pass、阴影和变体。每加一类数据，都要问三个问题：数据从哪里来，在哪个阶段使用，会增加哪类成本。

不宜一开始就把光照、阴影、透明、雾效、描边和大量关键字塞进同一个示例。Shader 一旦失去最小闭环，调试成本会显著上升。

常见误判

这类最小 Shader 经常暴露一些基础误判。把它们提前排掉，有助于后续复杂效果的排查。

Alpha 不透明

_Color 是 RGBA，但本文 Pass 没有写 Blend，也没有改 ZWrite。所以 _Color.a = 0.2 不会让物体变透明。透明需要改变渲染队列、混合状态、深度写入策略和排序预期，例如：

1
2
3

Tags { "RenderType"="Transparent" "Queue"="Transparent" }
Blend SrcAlpha OneMinusSrcAlpha
ZWrite Off

这不是本文示例的一部分，因为透明会引出排序、过度绘制和 Early-Z 退化问题，应该单独讲。

粉色错误

Unity 显示粉色通常表示 Shader 编译失败或当前管线不支持该 Shader。对本文示例，优先检查：

  • HLSLPROGRAM 和 ENDHLSL 是否成对。
  • #include "UnityCG.cginc" 是否拼写正确。
  • #pragma vertex vert 和函数名是否一致。
  • 当前项目是否真的在 Built-in 管线，或是否把它放进了 URP/HDRP 项目里强行使用。

颜色不变

材质面板改了颜色但物体不变，常见原因有：

  • Renderer 上绑定的并非正在修改的材质。
  • C# 改的是 sharedMaterial 或 material，但实际运行路径被 MaterialPropertyBlock 覆盖。
  • _Color 名字不一致。
  • 场景里还有另一个材质实例。
  • Frame Debugger 中实际执行的不是这个 Pass。

这类问题不宜靠猜测。打开 Frame Debugger，查看当前 Draw Call 的 Shader、Pass、Material 属性，通常能定位到主要原因。

验证

最后给一组可复现实验。不应只凭 Scene 视图里“好像有颜色”的印象，要沿着管线把关键点验证掉。

创建验证

  1. 新建 HelloColor.shader，写入本文完整代码。
  2. 新建 Material，Shader 选择 ShaderAllInOne/HelloColor。
  3. 新建 Cube 或 Sphere，把 Material 赋给 Renderer。
  4. 调整材质面板里的 Color，确认 Game 视图中物体颜色同步变化。
  5. 把相机移动到能看到物体的位置，避免把相机裁剪和 Layer Culling 误判成 Shader 问题。

Frame Debugger

在 Unity 中打开 Window > Analysis > Frame Debugger，启用后找到该物体的 Draw Call，检查：

  • Shader 名称是否是 ShaderAllInOne/HelloColor。
  • Pass 是否是本文唯一的 Pass。
  • Mesh 顶点输入中是否存在 POSITION。
  • _Color 当前值是否与材质面板一致。
  • Render Target 是否是相机颜色缓冲或预期中间纹理。

如果 Draw Call 不存在，优先查 Renderer 是否启用、Layer 是否被相机剔除、材质是否赋错、物体是否被视锥剔除。

破坏实验

做三个小破坏，能更快建立直觉：

  1. 把 #pragma fragment frag 改成不存在的函数名，Unity 应该报编译错误，材质变粉。
  2. 把 HLSL 中的 float4 _Color; 改成 float4 _Tint;，材质面板仍有 _Color，但片元读不到同名属性，结果会回退到未按预期绑定的值。
  3. 把 return _Color; 改成 return float4(0, 1, 0, 1);，如果物体仍不变绿，说明当前绘制根本没有走这个 Pass。

静态检查

写 Shader 时至少保留这张清单：

  • Properties 中的属性名和 HLSL 变量名一致。
  • #pragma vertex/fragment 指向真实存在的函数。
  • 顶点输出包含 SV_POSITION。
  • 片元输出包含 SV_Target。
  • 需要的 include 文件存在且属于当前管线。
  • Pass 状态与预期一致，尤其是透明、深度和混合。

小结

Hello-Color 的价值不在视觉复杂度，而在它把顶点片元 Shader 的主干压到了最低复杂度：

  • ShaderLab 负责外壳、属性、SubShader 和 Pass。
  • HLSLPROGRAM 负责声明 GPU 可编程阶段。
  • #pragma vertex/fragment 把入口函数绑定到顶点和片元阶段。
  • appdata 从 Mesh 读取顶点位置。
  • v2f 把顶点阶段输出交给固定功能阶段和片元阶段。
  • SV_POSITION 让硬件知道三角形如何投影和光栅化。
  • SV_Target 让片元函数输出候选颜色。
  • _Color 从 Material 进入本次 Draw Call 的 GPU 常量数据。

真正掌握这段代码后，再看纹理、光照、法线、透明和后处理，就不是在背 Shader 语法，而是在给这条最小数据流增加新的输入、计算和输出规则。
