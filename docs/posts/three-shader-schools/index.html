顶点-片元、Surface、Compute 不是三种“高级程度”不同的语法，而是三种不同的 GPU 程序入口。入口选错，后续代码再完整，也只是在错误的执行模型里打补丁。

总览

写 Unity Shader 时最容易混淆的一点，是把所有 GPU 代码都叫成“Shader”，然后用同一套直觉判断它们。实际工程里，顶点-片元着色器、Surface着色器、Compute 解决的是三类不同问题：谁负责画、谁负责生成光照代码、谁负责处理大批量数据。

三种入口

  • 顶点-片元 Shader：贴着硬件图形管线写。需显式声明 Vertex 和 Fragment 程序，自行组织顶点输出、片元输入、Pass 状态、深度、混合和渲染队列。
  • Surface Shader：Built-in 管线里的高层材质写法。开发者描述表面属性，Unity 的 Surface 编译器代为生成实际的顶点/片元程序和若干光照 Pass。
  • Compute Shader：绕开传统绘制管线。它不涉及 Mesh、三角形和光栅化，而是按线程组并行读写 Buffer 或 Texture。

不是等级

这三者没有“谁替代谁”的单线演进关系。更准确的判断口径是：

问题更常见的入口判断理由
我要控制顶点、片元、深度、透明和 Pass顶点-片元它直接绑定图形管线阶段，控制粒度最高
我要在 Built-in 里快速写受光材质Surface它能自动生成 Forward、Deferred、Shadow 等 Pass
我要做 GPU 剔除、粒子模拟、图像处理或写 DrawArgsCompute目标是数据，不是片元颜色

先问“输出是什么”。输出到 Render Target 的候选颜色，通常是图形管线问题；输出到 Buffer、RWTexture 或间接绘制参数，通常是 Compute 问题。

顶点片元

顶点-片元是最贴近 GPU 图形管线的写法。它的核心不是“手写更多代码”，而是把每个阶段的契约显式写清楚：顶点必须输出裁剪空间位置，光栅化器会插值 varyings，片元再计算候选颜色，最后交给深度、模板和混合规则。

数据流

一次最小绘制大致是这条路径：

flowchart LR
  A[Mesh 顶点] --> B[Vertex Shader]
  B --> C[裁剪空间位置 SV_POSITION]
  B --> D[Varyings: UV / Normal / WorldPos]
  C --> E[装配 / 裁剪 / 光栅化]
  D --> E
  E --> F[Fragment Shader]
  G[Texture / Constant Buffer / Light Data] --> F
  F --> H[Depth / Stencil / Blend]
  H --> I[Color Target]

Vertex 阶段的最低要求，是把顶点变换到裁剪空间。Fragment 阶段的最低要求，是输出一个颜色或多个 Render Target。它们中间的插值、透视校正和覆盖测试由固定功能硬件完成。

Shader "Demo/MinimalVertexFragment"
{
    Properties
    {
        _BaseMap ("Base Map", 2D) = "white" {}
        _Tint ("Tint", Color) = (1, 1, 1, 1)
    }

    SubShader
    {
        Tags { "RenderType"="Opaque" "Queue"="Geometry" }

        Pass
        {
            HLSLPROGRAM
            #pragma vertex vert
            #pragma fragment frag

            #include "UnityCG.cginc"

            sampler2D _BaseMap;
            float4 _BaseMap_ST;
            float4 _Tint;

            struct appdata
            {
                float4 vertex : POSITION;
                float2 uv : TEXCOORD0;
            };

            struct v2f
            {
                float4 pos : SV_POSITION;
                float2 uv : TEXCOORD0;
            };

            v2f vert(appdata v)
            {
                v2f o;
                o.pos = UnityObjectToClipPos(v.vertex);
                o.uv = TRANSFORM_TEX(v.uv, _BaseMap);
                return o;
            }

            float4 frag(v2f i) : SV_Target
            {
                return tex2D(_BaseMap, i.uv) * _Tint;
            }
            ENDHLSL
        }
    }
}

这段代码没有自动光照、阴影、GI 或额外 Pass。它的优势也正来自这里：所有成本都暴露在外，所有状态都可被显式管理。

适用面

顶点-片元适合这些场景：

  • Unlit、特效、扭曲、溶解、描边、护盾、水面、风场顶点动画。
  • URP/HDRP 里需要明确控制 Pass、Render Queue、Stencil、Depth、Blend 的效果。
  • 自定义 G-Buffer、替代光照模型、屏幕空间后处理、调试视图。
  • 项目自研 HLSL include，需要跨多个材质复用数学函数和采样规则。

它的代价也很明确：需要亲自处理管线契约。透明排序、ShadowCaster、DepthOnly、Meta Pass、SRP Batcher 常量布局、变体数量，都会成为工程成本。

顶点-片元不是“低级写法”，它是最直接的控制面。写得好时干净、可测、可迁移；写得散时，每个材质都会成为一套独立的、不可复用的管线。

Surface

Surface Shader 是 Unity Built-in Render Pipeline 的高层材质写法。它把“我想让这个表面如何响应光照”抽成 surf 函数，然后由 Unity 生成实际的 Vertex、Fragment 和一组渲染路径相关 Pass。

编译器

Surface 的关键不是语法糖，而是代码生成。开发者写的是：

1

#pragma surface surf Standard fullforwardshadows

Unity 看到这行后，会根据光照模型、渲染路径、阴影、GI、透明和可选指令，生成实际参与绘制的 Shader 程序。也就是说，Surface 表面上只写了一个 surf，背后却可能生成多个 Pass 和大量变体。

Shader "Demo/SurfaceStandard"
{
    Properties
    {
        _MainTex ("Albedo", 2D) = "white" {}
        _Metallic ("Metallic", Range(0, 1)) = 0
        _Smoothness ("Smoothness", Range(0, 1)) = 0.5
    }

    SubShader
    {
        Tags { "RenderType"="Opaque" }

        CGPROGRAM
        #pragma surface surf Standard fullforwardshadows
        #pragma target 3.0

        sampler2D _MainTex;
        half _Metallic;
        half _Smoothness;

        struct Input
        {
            float2 uv_MainTex;
        };

        void surf(Input IN, inout SurfaceOutputStandard o)
        {
            fixed4 c = tex2D(_MainTex, IN.uv_MainTex);
            o.Albedo = c.rgb;
            o.Metallic = _Metallic;
            o.Smoothness = _Smoothness;
            o.Alpha = c.a;
        }
        ENDCG
    }

    FallBack "Diffuse"
}

这段代码的价值在于少写了大量光照样板。它会接入 Built-in 的 Standard 光照模型，并根据需要处理 Forward、Deferred、阴影和 Meta 等路径。

隐性成本

Surface 的问题也来自代码生成。代码少写了，但成本也少看见了：

  • Pass 数量和变体可能比直觉更多。
  • 自定义光照、透明、深度、模板、特殊 Pass 会越来越绕。
  • 生成代码和手写逻辑之间隔着编译器，调试粒度更粗。
  • 它是 Built-in 体系的工具，不是 URP/HDRP 的主路径。

Unity 文档对兼容性说得很直接：Surface Shader 属于 Built-in Render Pipeline；URP、HDRP 和自定义 SRP 的兼容项不是 Surface。新项目如果已经确定走 SRP，不宜把 Surface 当作主力架构。

Surface 适合维护 Built-in 旧资产、快速写受光材质、讲解传统光照路径。它不适合作为新 URP/HDRP 项目的长期 Shader 基座。

扩展性说明：Surface 旧资产怎么留后路

如果项目里已经有大量 Surface Shader，不必急着机械重写。先把可迁移资产分层。

第一层是材质参数契约：统一 _BaseMap、_BaseColor、Normal、Metallic、Smoothness、Mask 通道含义。只要参数层混乱，迁移到 URP/HDRP 时就会变成逐材质考古。

第二层是公共函数契约：把风场、溶解、受击闪白、MatCap、Ramp 采样、法线解码放到项目自己的 include 里。Surface 只负责调用这些函数并填充 SurfaceOutput，业务规则不应写死在 surf 里。

第三层是渲染功能契约：记录每个效果依赖哪些 Pass、是否写深度、是否投阴影、是否需要 Meta Pass、是否参与 Lightmap。迁移时才能把它翻译成 URP Pass、Renderer Feature、Shader Graph Custom Function 或 HDRP Custom Pass。

Compute

Compute Shader 的思维模型需要切换。它不是“更强的片元 Shader”，也不是“可以随意写内存的通用入口”。它的基本单位是线程组，输入输出是 Buffer 或 Texture，核心工作是让 GPU 对大量规则数据并行执行同一段程序。

线程模型

典型 Compute 调用包含三层尺寸：

  • numthreads(x, y, z)：一个线程组里有多少线程。
  • Dispatch(groupX, groupY, groupZ)：启动多少个线程组。
  • SV_DispatchThreadID：当前线程在全局网格里的唯一坐标。

一个 1920x1080 的图像处理 Kernel，如果使用 numthreads(8, 8, 1)，常见 Dispatch 形状是：

1
2
3

int groupX = Mathf.CeilToInt(width / 8.0f);
int groupY = Mathf.CeilToInt(height / 8.0f);
compute.Dispatch(kernel, groupX, groupY, 1);

这不是为了取整。线程组大小会影响 Occupancy、共享内存、Wave/Warp 利用率、缓存局部性和平台兼容性。

// Invert.compute
#pragma kernel Invert

Texture2D<float4> _Source;
RWTexture2D<float4> _Result;
int2 _Size;

[numthreads(8, 8, 1)]
void Invert(uint3 id : SV_DispatchThreadID)
{
    if (id.x >= _Size.x || id.y >= _Size.y)
    {
        return;
    }

    float4 c = _Source[id.xy];
    _Result[id.xy] = float4(1.0 - c.rgb, c.a);
}

同步代价

Compute 的工程风险集中在数据流：

  • 结果是不是马上要被 Draw Pass 消费？
  • 是否需要资源状态转换或 Render Graph 依赖？
  • 是否需要 CPU 读回？
  • 写入是否存在竞争，是否需要原子操作或分块归约？
  • 是否会产生随机访问，压垮缓存和带宽？

Dispatch 后立刻 GetData 或同步读回，是 Compute 最常见的性能事故。GPU 本在并行执行，CPU 却同步阻塞等待结果。

选择路径

真正的选择不是“我喜欢哪种语法”，而是“这个功能在一帧里的数据流长什么样”。看不到输入、输出、消费者和同步边界，讨论 Shader 写法就是空转。

决策表

需求首选不选另两者的原因
水面顶点波动和片元高光顶点-片元Surface 难控特殊 Pass；Compute 输出还要回到绘制管线
Built-in 老项目里快速做受光角色材质Surface顶点-片元要补大量光照 Pass；Compute 不负责材质光照
GPU 粒子模拟Compute顶点-片元不适合维护粒子状态；Surface 不对口
屏幕空间描边顶点-片元或 Compute轻量全屏效果用片元；复杂多步边缘检测可用 Compute
GPU Driven 可见性剔除Compute输出是可见列表和 DrawArgs，不是颜色
URP/HDRP 自定义材质主路径顶点-片元或 Shader GraphSurface 不兼容 SRP 主路径

Mermaid 流程

这张流程图是我在项目里最常用的判断顺序：

flowchart TD
  A[要实现一个 Shader 功能] --> B{输出是什么}
  B -->|颜色 / 深度 / GBuffer| C{需要自动接入 Built-in 光照吗}
  B -->|Buffer / RWTexture / DrawArgs| D[Compute Shader]
  C -->|是，且项目是 Built-in| E[Surface Shader]
  C -->|否，或项目是 URP/HDRP| F[顶点-片元 / Shader Graph]
  F --> G{是否需要特殊 Pass 状态}
  G -->|需要| H[手写 Vertex-Fragment Pass]
  G -->|不需要| I[Shader Graph 或轻量手写]
  D --> J{CPU 是否立刻读回}
  J -->|是| K[重新设计：AsyncGPUReadback 或延迟消费]
  J -->|否| L[让后续 Draw / Dispatch 消费结果]

选择入口时，不要从语法熟悉度出发。从输出目标和消费者出发，工程路径会干净很多。

工程边界

一个长期项目不会只用一种 Shader 写法。真实情况通常是：材质主路径走 Shader Graph 或手写顶点-片元，少量旧 Built-in 资产仍然是 Surface，GPU 剔除、粒子和后处理中的部分步骤走 Compute。关键不是消灭差异，而是把差异关在正确边界里。

公共层

我更倾向于把 Shader 体系拆成四层：

  1. 数学层：颜色空间、法线解码、噪声、Ramp、BRDF 辅助函数。
  2. 材质层：贴图通道约定、材质参数命名、关键字策略。
  3. 管线层：Built-in、URP、HDRP 或自定义 SRP 的 Pass、include、宏和常量缓冲适配。
  4. 功能层：描边、溶解、风场、GPU 粒子、剔除、后处理等具体效果。

数学层和材质层应该尽量稳定。管线层可以薄适配。功能层负责组合，而不是把所有细节复制进每个 Shader 文件。

变体控制

三种流派都会制造变体压力，只是形态不同：

  • 顶点-片元：multi_compile、Pass 数量、LightMode、渲染队列。
  • Surface：Unity 自动生成的光照路径和选项变体。
  • Compute：Kernel 数量、线程组尺寸、资源格式和平台分支。

变体不是越少越好，而是必须可解释。每一个关键字都应该回答两个问题：它解决了什么真实画面差异？它是否值得占用构建时间、包体和运行时内存？

扩展性说明：三种写法如何共存

把顶点-片元、Surface、Compute 放进同一个项目时，风险在于让玩法、渲染和资源格式互相穿透。

更稳的方式是让玩法代码只依赖功能接口，比如“角色受击闪白”“写入可见实例列表”“执行边缘描边 Pass”。接口后面可以是手写 HLSL、Surface 旧资产、Renderer Feature、Render Graph Compute Pass，玩法侧不应该知道细节。

资源层也要留边界。Compute 输出的 Buffer 如果要被绘制消费，命名、stride、生命周期、写入时机要有文档；顶点-片元要消费的纹理，如果来自 Compute，就要明确格式、颜色空间、mipmap、是否可随机写。

迁移边界更现实：Surface 旧资产不要直接和新 SRP 功能互相调用。先把公共采样和材质函数抽出来，再分别做 Built-in Surface 适配和 URP/HDRP HLSL 适配。这样迁移不是重写效果，而是重写薄薄一层管线胶水。

调试口径

Shader 调试难，难点不在语法，而在问题可能发生在不同阶段。先定位阶段，再改代码。

现象定位

现象优先怀疑检查方式
模型消失或位置乱飞Vertex 输出、矩阵、Cull、Clip SpaceFrame Debugger、RenderDoc Mesh Viewer
贴图拉伸或采样错误UV、插值、Sampler、纹理导入输出 UV 调试色
透明顺序错Queue、ZWrite、Blend、排序Frame Debugger 看绘制顺序
阴影缺失ShadowCaster Pass、LightMode、RenderType检查 Pass 和 Cast Shadows
Compute 结果偶发错越界、竞争写、同步、资源状态小尺寸数据、GPU Capture、关闭异步路径
帧时间异常片元、带宽、变体、同步读回Unity Profiler、GPU Profiler、RenderDoc

验证顺序

我会按这个顺序排查：

  1. 先用纯色输出验证对象是否进入正确 Pass。
  2. 再输出 UV、法线、深度或 ID，确认数据流没断。
  3. 对 Compute，把输入规模缩小到 8x8 或 16x16，先证明线程坐标和边界判断正确。
  4. 最后再打开光照、阴影、透明、后处理和平台关键字。

排查时不宜同时改光照公式、Pass、Queue、关键字和 Compute 数据布局。一次只移动一个变量，调试才有因果关系。

总结

三大 Shader 流派可以压成三句话：

  • 顶点-片元：控制图形管线，适合需要明确 Pass、状态和渲染阶段的效果。
  • Surface：Built-in 的材质生成器，适合旧项目和快速受光材质，但不适合作为新 SRP 主线。
  • Compute：GPU 并行数据处理入口，适合 Buffer、Texture、剔除、模拟和后处理中的大规模规则计算。

真正成熟的 Shader 架构，不是把所有东西统一成一种写法，而是让每种写法只承担自己擅长的责任。顶点-片元负责画，Surface 负责维护 Built-in 存量材质，Compute 负责产出数据；公共数学、材质参数和资源契约单独沉淀。这样项目才有扩展性，也才经得起管线迁移和平台压测。
