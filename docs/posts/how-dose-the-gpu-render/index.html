如果把一帧画面看作最终成像结果，那么 CPU 更像调度者，负责决定这一帧要绘制哪些对象、按何种顺序提交；GPU 则是一套面向吞吐量优化的并行处理器，负责将三角形、材质、光照、纹理与后处理转换为屏幕上的像素。

总览

先把一帧视作一条流水线，而非一张静态图像。CPU、GPU 与显示器各负责其中一段，真正需要厘清的是它们之间的衔接关系。

三段系统

一帧并不是“游戏逻辑算完后直接呈现在屏幕上”。更准确地说，它依次经过三段系统：

  • CPU 侧：运行游戏逻辑、动画、物理计算，执行视锥剔除与渲染排序，最终生成渲染命令。
  • GPU 侧：执行图形管线，将顶点、三角形与材质数据转换为颜色缓冲中的像素。
  • 显示侧：显示控制器按固定时序读取最终图像，显示面板将数字颜色转换为光信号。

异步接力

这一过程有一个常被忽视的特征：CPU 与 GPU 通常并非严格串行等待。现代引擎会将命令排入命令队列，GPU 处理已提交或当前批次的命令，显示器随后读取已完成的后缓冲。帧延迟、VSync、双缓冲与三缓冲等机制，本质上都在处理这条流水线上的同步问题。

分析一帧不应只关注 GPU。CPU 端命令排队、GPU 端命令执行、显示器端图像扫描，任一环节出现瓶颈，最终都会反映为帧时间波动。

GPU 本质

GPU 并不是大量小型 CPU 的简单堆叠，而是一台面向吞吐量优化的并行计算设备。它依赖大量计算单元、分层缓存与固定功能模块，并行执行大量规则且重复的小型任务。

Die 结构

一张独立显卡大致由以下模块组成：

  • GPU Core：大量并行计算单元，用于执行顶点着色器、片元着色器、Compute Shader 等程序。
  • 显存 VRAM：存放纹理、网格、常量缓冲、渲染目标、深度缓冲等数据。
  • 缓存层级：包括 L1/L2 Cache、纹理缓存、常量缓存，用于降低访问显存所带来的时延成本。
  • 固定功能单元：光栅化器、纹理采样器、ROP/Blend 单元、显示控制器等。
  • 总线接口：PCIe 或 SoC 内部总线，用于与 CPU 及主存交换数据。

GPU 的优势并非来源于“单核性能突出”，而是来源于对海量小任务的并行处理能力。图形渲染天然契合这一模型：一个模型包含大量顶点，一张屏幕包含数百万像素，每个顶点与像素都可以在较大程度上独立计算。

上面这张 Die 图比抽象框图更具参考价值：从中可以直接看到 GPU 并非一整块均匀的“算力海”。大面积区域被 GPC/TPC 这类图形计算簇占据，中间是面积巨大的 L2 Cache，两侧与顶部为显存 PHY，底部还包含 PCIe、显示控制器等 I/O 逻辑。渲染一帧时，顶点处理、片元计算、纹理采样、深度混合、显存读写都会在这些不同区域之间流动。

SIMT 模型

GPU 的执行模型通常可以概括为 SIMT：Single Instruction, Multiple Threads。硬件将一批线程编为一组执行单位，NVIDIA 一般称为 Warp，AMD 一般称为 Wavefront。同一组线程在同一时刻执行同一条指令，但作用于不同的顶点、片元或纹理坐标。

该模型的强项在于吞吐能力，而非单任务延迟。一次显存访问可能需要等待上百个周期，GPU 不会空等，而是立即切换到另一组已就绪的 Warp。只要活跃 Warp 数量充足，访存延迟便可被并行执行掩盖。反之，若 Shader 中分支发散严重，或活跃线程数量不足以填满硬件，GPU 的并行优势便难以充分发挥。

SIMT 和 SIMD 的差别

SIMD 更接近“一条指令直接作用于一组数据”的模式，常见于 CPU 向量指令。SIMT 则允许程序员以看似独立的线程编写程序，再由硬件将这些线程编组执行。

其优势在于编程模型接近普通线程模型，劣势则在于：同一 Warp 内若部分线程走 if 分支、另一部分走 else 分支，硬件通常需要分批执行两条路径。这就是分支发散会降低吞吐的根本原因。

基本术语

以下关键术语需要先明确：

  • Shader：运行于 GPU 上的小程序。顶点着色器处理顶点，片元/像素着色器处理像素候选点，Compute Shader 处理更通用的并行任务。
  • Warp/Wavefront：GPU 将一组线程编为一组同时执行。NVIDIA 一般称为 Warp，AMD 一般称为 Wavefront。同一组线程共享一条指令流，因此分支发散会降低执行效率。
  • 带宽：显存吞吐能力。许多渲染瓶颈并非来自算力不足，而是来自数据搬运能力不足。
  • 固定功能管线：部分图形步骤使用频率高、规则固定，因此被实现为专用硬件，例如光栅化、纹理过滤、深度测试、混合。

CPU 发令

CPU 不负责逐像素绘制图像，它负责组织场景、筛选可见对象、录制渲染命令。GPU 在接收到这些命令后，才真正展开大规模并行计算。

主循环

在引擎中，一帧通常从 CPU 的主循环开始：

  1. 输入采样：读取键盘、鼠标、手柄、触摸。
  2. 游戏逻辑：更新角色状态、AI、技能、粒子、UI。
  3. 物理与动画：骨骼、碰撞、刚体、布料等。
  4. 可见性判断：视锥剔除、遮挡剔除、LOD 选择。
  5. 渲染排序：不透明物体、透明物体、阴影、后处理、UI。
  6. 命令提交：把 Draw Call、资源绑定、渲染状态写入命令缓冲。

CPU 不会逐像素计算颜色。它仅向 GPU 提交一条指令：“使用该 mesh、该 material、相应纹理与常量，向指定 Render Target 发起一次绘制。”

图形管线

实时渲染最核心的处理路径称为 Graphics Pipeline。不同 API 的命名略有差异，Direct3D、OpenGL、Vulkan、Metal、Unity SRP 都遵循这一主干。

该管线最重要的设计思想是：由可编程阶段负责灵活性，由固定功能阶段负责吞吐量。Shader 可控制顶点与像素的计算方式，但三角形覆盖哪些像素、深度如何比较、颜色如何混合等高频规则通常交由硬件处理。

Draw Call 并不等同于“绘制一个物体”，它对应一次命令提交。一次 Draw Call 可以绘制大量实例，也可能仅绘制一个小型 UI 元素。

顶点到片元

管线前半段解决“顶点位于何处”的问题。顶点首先被变换到相机空间与屏幕空间，三角形随后由硬件拆分为片元。

顶点变换

模型文件中的顶点通常处于 模型空间。例如一个角色的鼻尖坐标可能是 (0, 1.7, 0.2)，该坐标仅描述其相对于角色自身原点的位置。要将其绘制到屏幕上，需要依次经过一串空间变换：

顶点着色器最典型的工作就是矩阵乘法：

1

float4 clipPos = mul(UNITY_MATRIX_MVP, float4(positionOS, 1.0));

其中 MVP 可分解为：

  • M / Model Matrix：模型空间到世界空间。决定角色所在位置、朝向以及缩放比例。
  • V / View Matrix：世界空间到相机空间。将相机视为新坐标系的原点。
  • P / Projection Matrix：相机空间到裁剪空间。透视投影使远处物体更小。

顶点阶段并不会将模型转换为完整的表面。它仅处理离散的顶点，并将每个顶点附带的数据向下传递，例如 UV、法线、切线、顶点色、骨骼权重。将三角形展开为像素候选点的任务由下一阶段完成。

裁剪装配

GPU 接收顶点着色器输出后，会将索引缓冲中的三个顶点装配为三角形。随后依次执行以下步骤：

  • 背面剔除：若三角形背向相机，且材质开启了 Cull Back，则该三角形将被丢弃。
  • 视锥裁剪：超出相机裁剪空间的部分会被切除。
  • 透视除法：将 clipPos.xyzw 转换为 NDC = clipPos.xyz / clipPos.w。
  • Viewport 映射：将 [-1, 1] 范围的 NDC 映射至屏幕像素坐标。

之所以将三角形作为实时渲染的基本单位，是因为三角形必然共面，插值简单，硬件实现稳定。四边形与多边形最终也会被拆解为三角形。

光栅化

光栅化是整条管线中最重要的固定功能阶段。它回答一个基础问题：该三角形覆盖了屏幕上的哪些采样点？

被覆盖的采样点将生成 Fragment。Fragment 尚未成为最终像素，它仅是一个候选像素，附带插值后的属性，例如 UV、法线、世界坐标、颜色。

此处有两个常见误区：

  • 片元并不等同于像素：一个像素可能对应多个采样点，尤其在开启 MSAA 时；一个片元也可能因深度测试失败而被丢弃。
  • GPU 并非逐线绘制或逐区域填充：它是在规则采样网格上进行覆盖测试、属性插值与并行计算。

插值也不是简单的屏幕空间线性平均。透视投影会改变远近关系，若直接在屏幕空间对 UV 做线性插值，远处纹理会出现明显扭曲。因此硬件会执行 perspective-correct interpolation，利用裁剪空间的 w 分量对顶点属性进行修正，保证纹理与法线随透视关系正确变化。

为什么要透视校正

投影完成后，屏幕空间中的线性距离已不再等同于三维空间中的线性距离。近处相同长度的线段会占据更多像素，远处则占据更少像素。

若 UV 在屏幕空间直接线性插值，纹理会像贴在可变形的橡皮表面上一样出现错误拉伸。硬件利用每个顶点的 1/w 进行校正，使插值结果重新符合透视投影关系。

片元到缓冲

管线后半段解决“片元对应何种颜色、能否保留到缓冲”的问题。片元先计算候选颜色，再经过深度、模板与混合规则，最终写入渲染目标。

片元着色

Fragment Shader（也常称为 Pixel Shader）负责计算片元颜色。一个典型 PBR 材质大致会完成以下步骤：

  • 使用 UV 采样 Albedo、Normal、Metallic、Roughness、AO 等纹理。
  • 将法线从切线空间变换到世界空间。
  • 读取光源、阴影、环境贴图、反射探针。
  • 计算漫反射、镜面反射、菲涅尔项、微表面分布。
  • 输出颜色、透明度，或写入 G-Buffer。

在 Forward 渲染路径下，片元着色器通常直接输出最终颜色。在 Deferred 渲染路径下，它可能先输出 G-Buffer，例如 BaseColor、Normal、Roughness、Depth，再由后续 Lighting Pass 统一计算光照。

片元阶段通常是性能热点，因为屏幕像素数量巨大。1920×1080 约 207 万像素，3840×2160 约 829 万像素。若场景中存在大量半透明、阴影、多光源、后处理与过度绘制，同一屏幕位置可能被多次计算。

输出合并

片元着色器输出颜色后，并不会立即写入屏幕。需要经过输出合并阶段，常见规则包括：

  • Depth Test：比较当前片元深度与深度缓冲中的值，决定谁离相机更近。
  • Depth Write：当前片元通过深度测试后，是否更新深度缓冲。
  • Stencil Test：利用模板值进行区域遮蔽，例如描边、传送门、UI 裁剪。
  • Blending：将源颜色与目标颜色混合，用于实现透明、粒子、加法光效等效果。
  • Color Mask：控制是否写入 RGB/A 中的某些通道。

这正是不透明物体通常先绘制、透明物体后绘制的原因：

  • 不透明物体可以写入深度，后方的远处片元会被 Early-Z 提前丢弃。
  • 透明物体需要混合，通常不能简单写入深度，且一般需要按从远到近的顺序排序。

此处还存在一项收益很高的优化，称为 Early-Z：若片元在深度测试中注定失败，GPU 可在片元着色器执行前将其丢弃，从而节省昂贵的纹理采样与光照计算。但该优化有其前提条件：Shader 不能随意改写深度，也不能大量使用 discard 破坏硬件对深度的提前判断。

透明物体、discard、手写深度都会削弱 Early-Z 的发挥空间。片元瓶颈严重时,建议优先检查这些环节,比单纯降低贴图精度更可靠。

这里有几个关键细节：

  • 触发条件:深度测试默认 ZTest LEqual 且 ZWrite On,FS 不改写深度、不使用 discard。一旦任意一条不满足,硬件只能把深度测试推迟到 FS 之后,变成 Late-Z。
  • 节省的不是”测试本身”,而是 FS:Early-Z 通过测试的片元仍要执行 FS;只是被提前判定为”看不见”的片元,会跳过整段着色计算。在 4K 屏、复杂材质、半透明粒子这些场景,单帧可以省下数百万次纹理采样与光照计算。
  • 与绘制顺序协同:不透明物体先写深度,它后面的物体的远端片元才能在 Early-Z 阶段被丢。透明物体因为要混合,通常 ZWrite Off,不参与 Early-Z。
  • 不是所有 GPU 都一样:移动端 Tile-Based GPU 出于带宽考虑,有时会反过来要求 FS 写入深度,这时驱动和编译器会自动调整顺序。

帧缓冲

GPU 最终写入的目标并不是显示器本身，而是显存中的缓冲区。常见缓冲包括：

  • Color Buffer：颜色缓冲，SDR 一般为 RGBA8，HDR 一般为 RGBA16F。
  • Depth Buffer：深度缓冲，常见 24-bit 或 32-bit。
  • Stencil Buffer：模板缓冲，常与深度共用存储格式。
  • G-Buffer：在延迟渲染中用于保存法线、材质参数等中间数据。
  • Motion Vector：用于 TAA、Motion Blur、DLSS/FSR/XeSS 等特性。
  • Intermediate Render Targets：Bloom、SSAO、SSR、后处理链所需的中间纹理。

因此“一帧”在引擎内部往往对应一组纹理，而不仅是一张图。最终可被显示器读取并扫描输出的，通常是交换链中的某一个后缓冲。

后处理的本质是“将上一张纹理作为输入，再绘制一张全屏四边形”。Bloom、色调映射、景深、运动模糊、抗锯齿、颜色分级，本质上都属于屏幕空间处理。

显示输出

GPU 绘制完成后，图像仍只是显存中的数据。Present、交换链与显示扫描机制负责将这块数据转换为屏幕上的光信号。

交换链

应用程序通常不会直接写入前台正在显示的缓冲，而是写入 Back Buffer。绘制完成后调用 Present，将后缓冲交给显示系统。常见模式包括：

  • 双缓冲：一个前缓冲用于显示，一个后缓冲用于绘制。Present 时交换前后缓冲。
  • 三缓冲：增加一个缓冲，降低 CPU/GPU 因等待显示刷新而停顿的概率，但可能增加输入到显示之间的延迟。
  • VSync：等待垂直刷新边界再 Present，避免画面撕裂。
  • VRR：可变刷新率，显示器配合 GPU 改变刷新间隔，以降低撕裂与卡顿。

画面撕裂的产生原因较为直接：显示器正在从上到下扫描旧帧时，前缓冲中途被替换为新帧，导致屏幕上半部分与下半部分来自不同帧。

同步策略

VSync 的核心机制是仅允许在垂直空白期交换缓冲。60 Hz 显示器每 16.67 ms 提供一次交换窗口，若 GPU 在 16.67 ms 内完成绘制，可稳定输出 60 FPS；若耗时 18 ms，便错过当前 VBlank，只能等待下一次交换窗口，实际帧率会直接掉至 30 FPS 附近。这并非 GPU 只能绘制 30 帧，而是同步策略将不完整的中间结果按住所致。

三缓冲的思路是额外准备一个后缓冲，使 GPU 在等待显示扫描期间仍有空间继续绘制。该方案可缓解 VBlank 等待造成的流水线空泡，但可能增加输入到显示之间的排队延迟。G-Sync、FreeSync 等 VRR 技术则反向让显示器跟随 GPU 的完成节奏刷新，目标是在撕裂、卡顿与等待延迟之间取得更优平衡。

为什么会掉到 30 FPS

以 60 Hz 为例，显示器每 16.67 ms 扫描一帧。若 GPU 在 17 ms 才完成绘制，虽然仅超出 0.33 ms，但已错过当前 VBlank，只能等待下一次交换窗口。

因此在传统双缓冲 + VSync 模式下，帧率经常并非平滑地从 60 FPS 降至 55 FPS，而是直接跌至 30 FPS 附近。三缓冲与 VRR 都在解决这一同步颗粒度问题。

屏幕发光

进入显示侧时，图像已是一块二维颜色数组。每个像素通常包含红、绿、蓝三个颜色分量。显示控制器按固定时序读取像素数据，并通过 HDMI、DisplayPort、eDP、MIPI 等接口传输至显示面板。

显示面板进一步将每个像素拆分为更细的 RGB 子像素：

  • LCD：背光常亮或分区控制，液晶层调节每个子像素的透光量，彩色滤光片形成 RGB。
  • OLED：每个子像素自发光，通过电流强弱控制亮度。
  • Mini LED / Micro LED：控制方式有所不同，但目标均是将 RGB 数字值转换为光强。

此外还涉及颜色空间问题：

  • GPU Shader 通常在线性空间进行光照计算，因为真实光照能量近似遵循线性叠加。
  • 普通 SDR 图像一般以 sRGB 编码保存与显示，其编码值并非线性亮度。
  • HDR 渲染会在高动态范围缓冲中累积亮度，再经 Tone Mapping 压缩至显示设备可表达的范围。
  • Gamma、色域、亮度与面板校准都会影响最终呈现在屏幕上的颜色。

因此，Shader 输出 float4(1, 0, 0, 1) 并不等同于肉眼看到一个具有绝对物理意义的红色。它只是渲染管线中的一个颜色编码值，后续还需经过色彩管理、显示传输函数与面板响应的处理。

Shader 中计算的是编码后的颜色值，并非人眼最终感知。线性空间、sRGB、HDR、Tone Mapping 混用时，颜色相关问题会变得复杂且容易出错。

最小例子

借助一个红色三角形，可以将前文介绍的硬件、管线、缓冲与显示机制串联起来。复杂场景仅是这条路径的密集重复，并插入更多 Pass。

十步流程

假设需要绘制一个红色三角形：

  1. CPU 创建顶点缓冲：包含三个顶点，位置分别为 A/B/C。
  2. CPU 绑定材质：使用一个简单的 Shader，输出红色。
  3. CPU 提交 Draw Call：通知 GPU 绘制 1 个三角形。
  4. Vertex Shader 运行 3 次：将 3 个顶点从模型空间变换到裁剪空间。
  5. 固定功能阶段完成三角形装配：判断朝向、裁剪、映射至屏幕。
  6. Rasterizer 扫描三角形覆盖的像素采样点，生成片元。
  7. Fragment Shader 运行多次：每个片元输出红色。
  8. Depth Test 判断红色片元是否比已有内容更近。
  9. Blend/ROP 将通过测试的颜色写入 Color Buffer。
  10. Present 后显示器读取 Color Buffer，RGB 子像素发光，人眼看到红色三角形。

以上即为 GPU 绘制一帧的基本骨架。复杂场景仅是这一流程的密集重复，并插入阴影、后处理、粒子、UI、反射、全局光照等额外 Pass。

现代管线

上文介绍的是经典光栅化管线。在真实项目中，该管线正在被若干新的处理路径扩展。

Mesh 管线

Mesh Shader 将传统 Input Assembler、Vertex Shader 与 Geometry Shader 的一部分职责改造为更接近 Compute 的可编程阶段。该方案适合实现 GPU-Driven Culling、LOD 选择与小批量图元生成，典型收益是降低 CPU 逐物体提交 Draw Call 的压力。

光追路径

Ray Tracing 走的是另一条路径：它并不从三角形覆盖像素出发，而是从像素发射光线，并在 BVH 加速结构中查找相交的三角形。RT Core 等专用硬件主要用于加速 BVH 遍历与射线-三角形求交，常见于反射、阴影、全局光照等难以用传统光栅化准确表达的效果。

Compute

Compute Shader 完全绕开固定图形管线，将 GPU 视作通用并行处理器使用。现代引擎将其用于剔除、粒子、后处理、Hi-Z、SSR、TAA、GPU Skinning，乃至整套可见性系统。

VRS

Variable Rate Shading 的思路较为直接：并非所有屏幕区域都需要 1×1 精度着色。运动模糊区域、远景、周边视觉区域可使用 2×2 或更粗的着色率换取吞吐，再将预算留给视觉重点区域。

性能定位

性能优化的核心并非简单删减效果，而在于先判断瓶颈位于 CPU、顶点、片元、带宽还是同步环节，再针对具体结构与资源进行修改。

找瓶颈

当一帧耗时超出预算，例如 60 FPS 下超过 16.67 ms，便会出现掉帧。常见瓶颈可按管线阶段进行定位：

瓶颈典型原因直觉判断
CPU BoundDraw Call 过多、脚本逻辑偏重、剔除效果差、主线程等待降低分辨率后 FPS 提升不明显
Vertex Bound顶点过多、蒙皮复杂、曲面细分开销大降低模型 LOD 后可获得明显收益
Fragment Bound分辨率较高、透明物体多、后处理重、光照复杂降低分辨率后 FPS 明显提升
Texture Bound纹理采样频繁、缓存命中率低、各向异性等级高减少采样次数或优化纹理布局效果显著
Bandwidth BoundMRT 较多、HDR 格式占用大、Render Target 读写频繁降低 RT 精度或减少 Pass 数量效果显著
SynchronizationCPU/GPU 相互等待、读回操作多、资源屏障过细Profiler 中出现长等待或 pipeline bubble

性能优化不能依赖直觉。正确的做法是先使用工具定位瓶颈，例如 Unity Profiler、Frame Debugger、RenderDoc、Xcode GPU Capture、Nsight、Radeon GPU Profiler。需要明确时间具体消耗在哪个 Pass、哪个 Draw Call、哪个 Shader，而非依赖主观判断删减效果。

未经 Profile 的优化近乎盲目。应先证明瓶颈所在，再修改代码；否则容易在画质降低的同时帧时间却未见改善。

对症优化

以下是若干工程实践中可直接采用的判断方向：

  • Draw Call 偏多：优先考虑 SRP Batcher、GPU Instancing、Indirect Draw、材质排序与静态合批，但不应为追求合批而牺牲剔除粒度。
  • Fragment 压力较大：先通过降低分辨率进行验证，再检查透明 Overdraw、后处理链、复杂 PBR 分支与阴影采样次数。
  • Bandwidth 接近上限：检查 Render Target 格式、MRT 数量、MSAA 设置、HDR 中间纹理、频繁的 Blit 操作以及不必要的全屏 Pass。
  • 同步等待较多：警惕 CPU 读回 GPU 数据、过细的资源屏障、Present 阻塞以及 Frame Debugger 中观察到的 pipeline bubble。

易混概念

Draw Call 不等于三角形数量

Draw Call 是一次命令提交。一次 Draw Call 可以绘制大量三角形，也可能仅绘制一个小型 UI 元素。过多 Draw Call 会增加 CPU 与驱动开销，但合批本身也有代价，会影响剔除、材质切换与内存布局。

Fragment Shader 调用次数不等于屏幕像素数

过度绘制、透明、MSAA、阴影与后处理都会使着色次数超过最终像素数。4K 屏幕包含约 829 万像素，但一帧中实际执行的片元计算次数可能远超该数量。

Framebuffer 不等同于显示器

Framebuffer 是显存中的渲染目标。显示器仅最终读取其中的某一个可展示缓冲。许多中间 Framebuffer 永远不会直接用于显示。

Shader 计算的是颜色编码，并非物理世界本身

PBR 仅是一套近似光与材质相互作用的模型。实时渲染始终在精度、速度与稳定性之间进行折中。

总结

GPU 绘制一帧并非依赖某一项单点技术，而是 CPU、GPU、显存、交换链与显示器多方协作的结果。理解这一链路后，后续的 Shader 优化与渲染管线分析才有统一的坐标系。

一句话

GPU 绘制一帧的过程，可以概括为：CPU 将场景分解为渲染命令，GPU 利用并行管线将顶点装配为三角形，再将三角形光栅化为片元，由 Shader 计算颜色，经深度测试与混合写入帧缓冲，最终由显示系统将二维颜色数组扫描至屏幕，驱动 RGB 子像素发光。

在真正理解这条链路之后，Shader、渲染管线、后处理、抗锯齿、性能优化将不再是零散的知识点。它们均对应这条流水线上的某一处理环节、缓存层级或计算阶段。
