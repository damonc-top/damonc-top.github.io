固定管线和可编程管线的差别，不是“老 API”和“新 API”的差别，而是渲染算法入口的位置变了：一部分原本只能配置的固定规则，变成了可以由 Shader 和引擎显式组织的 GPU 程序。

背景

我最早接触 Shader 时，很容易把它理解成“给模型上色的小程序”。这个理解没有错，但覆盖范围偏窄。Shader 真正重要的地方，不在于它能把一个球画成金属或玻璃，而在于它改变了图形管线的控制边界：一部分原本由 API 和硬件预设的渲染算法，变成了引擎和内容侧可以编写、替换、组合的 GPU 程序。

这条演进线很适合作为 Shader 学习的第二个坐标。上一篇我把一帧画面的路径拆成 CPU 发令、GPU 管线、片元着色、输出合并和显示扫描。这一篇关注另一个问题：为什么这条管线里有些阶段仍然是固定硬件，有些阶段却变成了可编程 Shader？

结论并不是“可编程一定更先进，固定管线一定落后”。更准确地说：

  • 规则稳定、吞吐极高的阶段适合固定功能硬件，例如光栅化、纹理过滤、深度测试、混合。
  • 表达变化多、艺术需求强的阶段适合可编程逻辑，例如顶点变形、材质计算、光照模型、后处理。
  • 现代渲染管线是固定功能与可编程阶段的混合体，并不是把所有东西都改成 Shader。

这也是我理解 Unity Built-in、URP、HDRP、自定义 SRP 时很关键的一点：所谓“可编程管线”，不是抛弃硬件图形管线，而是在硬件允许的位置重写决策逻辑。

演进脉络

如果把这条演进线压缩成一张表，大致是这样：

阶段主要特征开发者能控制什么
固定功能管线API 提供矩阵、灯光、材质、纹理混合等固定状态调参数、切状态、提交 Draw
顶点/像素 Shader顶点处理和片元颜色计算开放成 GPU 程序顶点变换、材质、光照、纹理组合

固定管线

早期实时 3D API 提供的是 Fixed-Function Pipeline。开发者把顶点、矩阵、灯光、材质、纹理和一堆状态交给 API，硬件按照预定义路径完成变换、光照、纹理混合和输出。

从工程视角看，它像一台封装好的渲染机器：

1

设置矩阵 -> 设置灯光 -> 设置材质 -> 设置纹理 -> Draw

开发者能改很多参数，却不能改核心算法。比如顶点如何变换、光照如何计算、纹理如何组合，大部分都由 API 和硬件固定下来。

固定管线的问题不是“不能调参数”，而是核心算法入口不可替换。能调灯光颜色和能重写光照模型，是两种完全不同的控制粒度。

能力边界

固定管线的问题也很明显：它只能表达硬件和 API 预先想到的东西。

比如我想做一个非标准材质：

  • 顶点沿法线做风吹摆动。
  • 角色受伤区域按遮罩闪烁。
  • 法线贴图使用自定义解码方式。
  • 卡通光照需要阶梯式明暗边界。
  • 水面反射要混合深度雾、扰动和菲涅尔。

这些需求如果只靠固定状态组合，会很快撞墙。固定管线可以提供开关，但不能让我改掉“这盏灯到底怎么算”“这个顶点到底怎么动”“这个像素到底怎么混合纹理和光照”。

固定管线的本质限制是：它把渲染算法产品化了，却没有把渲染算法开放出来。

可编程阶段

可编程管线的关键，是把一部分固定算法替换成 GPU 上运行的小程序。最早被开放出来的是顶点和像素相关阶段，也就是后来最常接触的 Vertex Shader 和 Pixel Shader。

这不是简单增加两个功能点，而是把管线从“状态机”推进到了“程序执行模型”。

图里最值得注意的不是 Shader 变多了，而是固定功能仍然存在。光栅化、深度测试、混合这些阶段没有消失，因为它们太规则、太高频，交给专用硬件依旧划算。

顶点着色器

Vertex Shader 接管的是“每个顶点如何被处理”。最经典的工作是把模型空间顶点变换到裁剪空间：

1

float4 clipPos = mul(UNITY_MATRIX_MVP, float4(positionOS, 1.0));

但它的价值远不止矩阵乘法。只要是以顶点为粒度的计算，都可能放在这里：

  • 骨骼蒙皮。
  • 顶点动画。
  • 草、旗帜、布料的简化摆动。
  • GPU Instancing 的实例变换。
  • 程序化变形。
  • 向片元阶段传递自定义插值数据。

固定管线时代，顶点变换和光照大多是预设模型。Vertex Shader 出现后，引擎可以决定顶点到底携带哪些语义，如何进入裁剪空间，哪些中间数据要被插值给像素。

像素着色器

Pixel Shader 或 Fragment Shader 接管的是“每个片元如何计算候选颜色”。它让材质表达力获得直接提升。

一个极简片元着色器可能只是返回红色：

{

}

但真实项目里，它会组合纹理、法线、光照、阴影、反射、透明度和各种风格化规则：

1
2
3
4
5

float3 albedo = SAMPLE_TEXTURE2D(_BaseMap, sampler_BaseMap, uv).rgb;
float3 normalWS = normalize(input.normalWS);
float ndotl = saturate(dot(normalWS, _MainLightDirection));
float3 color = albedo * (_AmbientColor + _MainLightColor * ndotl);
return float4(color, 1.0);

这段代码很粗糙，但它揭示了算法入口的变化：光照公式不再只能选择 API 预设的几种模式，而是可以由 Shader 定义。PBR、卡通渲染、皮肤、头发、水、体积云、能量护盾，本质上都在利用这种可编程空间。

几何着色器

Geometry Shader 曾经被视为进一步开放管线的阶段。它位于图元装配之后，可以读取一个点、一条线或一个三角形，再输出新的图元。

它适合表达一些特殊需求：

  • 从点扩展出 billboard。
  • 生成简单轮廓线。
  • 调试法线和切线。
  • 做少量程序化图元扩展。

但在工业实践里，Geometry Shader 并没有成为主力路径。它的吞吐和硬件利用率经常不理想，在移动端也不总是友好。很多原本想放在 Geometry Shader 里的工作，后来被 Compute Shader、Mesh Shader、CPU 预处理或实例化方案替代。

这个阶段给我的启发是：管线阶段开放出来，不代表它一定适合承载高频主路径。架构设计还是要看硬件执行模型和数据规模。

“可编程”不等于“适合放主路径”。Geometry Shader、复杂分支、随机纹理访问都能写，但能写不代表能在目标平台稳定跑满帧率。

统一着色器

早期 GPU 常把顶点处理和像素处理做成不同硬件单元。问题是不同场景的负载比例并不稳定：有些场景顶点多，有些场景像素重，有些场景后处理占大头。

固定比例的硬件单元会造成浪费。比如顶点单元很闲、像素单元爆满时，空闲的顶点硬件不能承担像素任务。

统一着色器架构解决的是这个资源调度问题：把底层计算单元统一成通用 Shader Core，再由硬件和驱动把 Vertex Shader、Pixel Shader、Compute Shader 等任务调度到同一批执行资源上。

从引擎侧看，这带来几个直接影响：

  • GPU 更像一台通用并行处理器，而不是几个固定图形模块的拼装。
  • 不同 Pass 会竞争同一批执行资源，瓶颈分析必须看整体 GPU 时间。
  • Compute Shader 成为图形管线之外的关键工具。
  • GPU Driven 渲染、延迟光照、屏幕空间后处理、粒子模拟都能进入统一调度模型。

这也是现代渲染越来越依赖 GPU Profiler 的原因。仅凭“我这个 Shader 指令不多”很难判断成本，因为真实瓶颈可能在纹理采样、带宽、Occupancy、分支发散或 Render Target 读写。

Compute 扩展

Compute Shader 进一步打破了图形管线的形状。它不要求输入必须是顶点，也不要求输出必须是颜色。它只表达一件事：在 GPU 上启动大量线程，读写 Buffer 或 Texture。

这让 GPU 不再只是“画三角形的机器”，而是成为引擎里的并行数据处理器。

常见用途

我在理解现代管线时，会把 Compute Shader 用途分成几类：

  • 可见性与剔除：Frustum Culling、Occlusion Culling、Hi-Z Culling。
  • 数据生成：粒子模拟、程序化草、实例列表、间接绘制参数。
  • 屏幕空间效果：Bloom、SSAO、SSR、TAA、降噪、模糊。
  • 光照与聚类：Tiled Lighting、Clustered Lighting、光源列表构建。
  • 资源预处理：Mipmap 生成、前缀和、压缩/解压辅助、GPU Skinning。

它的价值不是“Compute 比 Pixel 更高级”，而是它避开了图形管线的固定输入输出约束。对于很多非绘制型任务，Compute 的数据布局和线程组织更直接。

工程代价

Compute 也带来更硬的工程问题：

  • 需要自己设计线程组大小和数据布局。
  • 需要处理资源状态、同步和内存屏障。
  • 随机读写可能造成缓存效率问题。
  • 不同平台能力差异更明显。
  • Debug 难度通常高于普通材质 Shader。

因此我不会把 Compute 当作万能优化按钮。它适合吞吐型、规则型、大批量的并行任务。如果数据规模小，或者频繁 CPU/GPU 同步，Compute 反而可能让系统更慢。

Compute 的风险通常不在“线程能不能跑”，而在数据布局、资源状态和同步。Dispatch 后立刻读回 CPU，常常会把并行吞吐变成同步等待。

从固定管线到可编程管线，变化最大的不是某个 Shader 语法，而是引擎分层方式。

它们不是为了把事情复杂化，而是因为可编程能力把原本封闭在驱动和硬件里的选择暴露了出来。自由度上升以后，工程系统必须承接复杂度。

Unity 视角

Unity 的演进也能放进这条线里理解。

Built-in 管线

Built-in Render Pipeline 给了开发者不少 Shader 编写空间，但渲染主流程本身很大程度仍由 Unity 控制。Forward、Deferred、ShadowCaster、LightMode、Surface Shader、GrabPass、内置光照宏，这些机制降低了上手成本，也把很多管线决策封装起来。

对中小项目来说，这种封装曾经很有效。但当项目需要自定义渲染顺序、定制光照结构、控制中间 Render Target、优化移动端带宽时，Built-in 的黑盒感会越来越强。

URP 和 HDRP

URP 和 HDRP 的关键变化，不只是“画质模板不同”，而是渲染管线变成了包和代码结构。

  • URP 偏向跨平台、可扩展和性能可控。
  • HDRP 偏向高端平台、物理一致性和复杂光照。
  • Shader Graph 降低了材质节点表达成本。
  • Renderer Feature / Custom Pass 提供了插入自定义渲染逻辑的位置。

这说明 Unity 已经不再把渲染管线只当成引擎内部实现，而是把它变成项目可以参与配置和扩展的系统。

using System.Collections.Generic;
using UnityEngine;
using UnityEngine.Rendering;

public sealed class MinimalRenderPipeline : RenderPipeline
{
    protected override void Render(
        ScriptableRenderContext context,
        List<Camera> cameras)
    {
        foreach (Camera camera in cameras)
        {
            if (!camera.TryGetCullingParameters(out var cullingParameters))
            {
                continue;
            }

            CullingResults cullingResults = context.Cull(ref cullingParameters);
            context.SetupCameraProperties(camera);

            var sortingSettings = new SortingSettings(camera);
            var drawingSettings = new DrawingSettings(
                new ShaderTagId("SRPDefaultUnlit"),
                sortingSettings);
            var filteringSettings = FilteringSettings.defaultValue;

            context.DrawRenderers(
                cullingResults,
                ref drawingSettings,
                ref filteringSettings);

            context.Submit();
        }
    }
}

这段代码不是完整生产管线，但它说明了 SRP 的思想：C# 负责组织相机、剔除、排序、Pass 和提交顺序；Shader 负责具体的 GPU 计算；底层图形 API 和硬件负责执行命令。

从这个角度看，SRP 是可编程管线在引擎层的一次延伸。Shader 让材质和顶点可编程，SRP 让渲染流程本身也变得可组织、可插拔、可裁剪。

性能逻辑

可编程能力会制造一种错觉：既然 Shader 可以自由编写，那效果就只是代码问题。实际工程里结论相反，Shader 越自由，越需要尊重硬件。

固定功能仍然值钱

很多固定功能阶段保留下来，是因为它们的效率太高：

  • 光栅化器能高速判断三角形覆盖。
  • 纹理单元能做过滤、寻址和缓存。
  • 深度测试和 Early-Z 能提前丢弃大量片元。
  • ROP/Blend 单元能高效完成颜色写入和混合。

如果一个效果可以顺着这些硬件路径走，通常比完全绕开它更稳。比如不透明物体优先写深度，让 Early-Z 工作；合理使用 Mipmap 和纹理压缩，让纹理单元发挥缓存优势；减少无意义的全屏 Blit，降低带宽压力。

不要为了“全可编程”绕开固定功能路径。光栅化、纹理过滤、Early-Z、ROP 都是硬件已经高度优化过的吞吐路径，能顺着走就不要人为拆解。

Shader 自由不是免费

Shader 的每一份表达力都要付账：

成本类型典型来源可能后果
ALU复杂数学、PBR、噪声、循环计算单元占满
Texture多贴图采样、高各向异性、随机采样纹理缓存压力和采样延迟
BandwidthMRT、HDR RT、MSAA、频繁 Blit显存带宽成为瓶颈
Divergence分支发散、材质逻辑差异大Warp/Wavefront 利用率下降
Register临时变量多、函数展开复杂Occupancy 下降
Variant关键字组合爆炸包体、编译时间、运行时切换成本上升

这张表比“少写几行 Shader”更有价值。优化不是追求代码短，而是判断当前 Pass 到底被哪种资源卡住。

管线级优化

可编程管线下，很多优化不在单个 Shader 里，而在 Pass 编排和资源生命周期里：

  • Shadow Pass 是否重复渲染了过多对象。
  • Depth Prepass 是否值得，还是增加了额外带宽。
  • G-Buffer 格式是否过宽。
  • 后处理链是否有不必要的中间 RT。
  • 透明物体排序是否导致大量 Overdraw。
  • Shader Variant 是否让材质批次碎裂。
  • Render Feature 是否破坏了原本可以合并的 Pass。

这也是我现在看渲染性能时更倾向从 Frame Debugger、RenderDoc、Profiler 入手的原因。Shader 只是局部，帧才是整体。

思维变化

固定管线到可编程管线的演进，最后改变的是思维方式。

从状态到程序

固定管线关注“打开哪个开关”。可编程管线关注“这段数据经过什么程序变成结果”。同样是材质，前者像参数表，后者像一段小型并行程序。

从单 Pass 到帧图

早期看渲染，容易聚焦在某个物体怎么画。现代渲染更应该看整帧如何组织：阴影、深度、G-Buffer、光照、透明、后处理、UI，每个 Pass 都在读写资源。

从 API 调用到数据流

Draw Call 只是入口。真正决定效率的是数据如何流动：顶点从哪里来，常量如何绑定，纹理如何采样，Render Target 如何读写，GPU 是否等待 CPU，Pass 之间有没有多余同步。

从效果实现到预算管理

可编程管线让很多效果都“能做”，但项目真正关心的是在目标平台预算内能不能稳定落地。移动端、主机、PC、VR 的约束完全不同，渲染架构必须服务于预算，而不是服务于炫技。

总结

固定管线不是失败的旧时代，它解决了早期实时 3D 最关键的问题：用稳定、封装、高吞吐的方式把三角形画到屏幕上。可编程管线也不是把一切推倒重来，它只是把最需要变化的阶段开放给 Shader 和引擎。

我会把这次演进总结成三句话：

  • 固定管线提供确定性：硬件和 API 预设好路径，开发者配置状态。
  • 可编程管线提供表达力：Shader 接管顶点、片元和通用并行计算。
  • 现代管线提供架构权：SRP、Render Graph、Compute、GPU Driven 让引擎重新组织整帧数据流。

真正理解 Shader，不只是会写 float4 frag()。更重要的是知道这段程序站在管线的哪个位置，前面输入什么，后面受哪些固定功能阶段约束，最终会消耗哪类 GPU 资源。

固定管线到可编程管线的演进，本质上是实时渲染从“硬件预设算法”走向“引擎组织数据流”的过程。Shader 是入口，但管线才是全局。
