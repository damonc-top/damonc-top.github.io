GPU 是怎么画一帧的：从显卡原理到屏幕成像

从 CPU 提交渲染命令开始，沿着 GPU 图形管线、光栅化、片元着色、帧缓冲、交换链和显示扫描，解释一帧图像如何被画到屏幕上。

如果把一帧画面想象成一张最终照片，那么 CPU 像导演，负责决定这一帧要拍谁、按什么顺序拍；GPU 像一座高度并行的图像工厂，负责把三角形、材质、光照、纹理和后处理变成屏幕上的像素。

总览

先把一帧看成一条流水线，而不是一张静态图片。CPU、GPU 和显示器各管一段，真正的问题是它们如何接力。

三段系统

一帧并不是“游戏逻辑算完后直接出现在屏幕上”。更准确地说，它经过了三段系统：

  • CPU 侧：游戏逻辑、动画、物理、剔除、排序、生成渲染命令。
  • GPU 侧：执行图形管线，把顶点、三角形和材质变成颜色缓冲。
  • 显示侧：显示控制器读取最终图像，屏幕把数字颜色变成光。

异步接力

这张图有一个反直觉点：CPU 和 GPU 通常不是严格串行等待的。现代引擎会把命令排进队列，GPU 消费上一批或当前批命令，显示器再读取已经完成的后缓冲。帧延迟、VSync、双缓冲、三缓冲，本质上都在处理这条流水线的同步问题。

看一帧不要只盯着 GPU。CPU 排队、GPU 执行、显示器扫描，任何一段被卡住，最终都会表现成帧时间抖动。

GPU 本质

GPU 不是很多个小 CPU 拼在一起，而是一台为吞吐量设计的并行机器。它用大量计算单元、缓存和固定功能模块，批量并行执行规则重复的小任务。

Die 结构

一张独立显卡大致由这些模块组成：

  • GPU Core：大量并行计算单元，用来跑顶点着色器、片元着色器、Compute Shader 等程序。
  • 显存 VRAM：存放纹理、网格、常量缓冲、渲染目标、深度缓冲等数据。
  • 缓存层级：L1/L2 Cache、纹理缓存、常量缓存，负责降低访问显存的时延成本。
  • 固定功能单元：光栅化器、纹理采样器、ROP/Blend 单元、显示控制器等。
  • 总线接口：PCIe 或 SoC 内部总线，用来和 CPU/主存交换数据。

GPU 强不是因为“单个核心特别聪明”，而是因为它能让成千上万个小任务一起跑。图形渲染天然适合这种模型：一个模型有很多顶点，一张屏幕有几百万像素，每个顶点、每个像素都可以在很大程度上独立计算。

上面这张 Die 图比抽象框图更有价值：它能直接看到 GPU 并不是一整块均匀的“算力海”。大面积区域被 GPC/TPC 这类图形计算簇占据，中间有巨大的 L2 Cache，两侧和顶部是显存 PHY，底部还有 PCIe、显示控制器等 I/O 逻辑。渲染一帧时，顶点、片元、纹理采样、深度混合、显存读写，都会在这些不同区域之间流动。

SIMT 模型

GPU 的执行模型通常可以理解为 SIMT：Single Instruction, Multiple Threads。硬件把一批线程捆成一个执行组，NVIDIA 常叫 Warp，AMD 常叫 Wavefront。同一组线程在同一时刻执行同一条指令，处理不同顶点、不同片元、不同纹理坐标。

这套模型的强项是吞吐，而不是低延迟。一次显存访问可能要等上百个周期，GPU 不会傻等，它会立刻切到另一个已经就绪的 Warp。只要场上有足够多的顶点和片元，延迟就被海量并发吞掉了。反过来，如果 Shader 里分支发散严重，或者线程数量不够填满硬件，GPU 的“核心很多”就会打折。

SIMT 和 SIMD 的差别

SIMD 更像“一条指令直接操作一组数据”，常见于 CPU 向量指令。SIMT 则让程序员写出看似独立的线程，硬件再把这些线程编组执行。

它的好处是编程模型更接近普通线程，坏处是同一个 Warp 里如果有人走 if 分支、有人走 else 分支，硬件往往要分批执行两条路径。这就是分支发散会降低吞吐的根本原因。

基本术语

几个关键词需要先卡住：

  • Shader：跑在 GPU 上的小程序。顶点着色器处理顶点，片元/像素着色器处理像素候选点，Compute Shader 处理更通用的并行任务。
  • Warp/Wavefront：GPU 会把一组线程捆在一起执行。NVIDIA 常叫 Warp，AMD 常叫 Wavefront。它们共享一条指令流，因此分支发散会降低效率。
  • 带宽：显存吞吐能力。很多渲染瓶颈不是算不动，而是数据搬不动。
  • 固定功能管线：某些图形步骤太常用、太规则，所以被做成专用硬件，例如光栅化、纹理过滤、深度测试、混合。

CPU 发令

CPU 不负责逐像素画图，它负责组织世界、筛选对象、录制命令。GPU 拿到这些命令后，才开始真正展开海量并行工作。

主循环

在引擎里，一帧通常从 CPU 的主循环开始：

  1. 输入采样：读取键盘、鼠标、手柄、触摸。
  2. 游戏逻辑：更新角色状态、AI、技能、粒子、UI。
  3. 物理与动画：骨骼、碰撞、刚体、布料等。
  4. 可见性判断：视锥剔除、遮挡剔除、LOD 选择。
  5. 渲染排序：不透明物体、透明物体、阴影、后处理、UI。
  6. 命令提交：把 Draw Call、资源绑定、渲染状态写进命令缓冲。

CPU 不会对每个像素亲自计算颜色。它只告诉 GPU：“用这个 mesh、这个 material、这些纹理和常量，向这个 Render Target 画一次。”

图形管线

实时渲染最核心的路径叫 Graphics Pipeline。不同 API 的命名略有差异，Direct3D、OpenGL、Vulkan、Metal、Unity SRP 都逃不开这条主干。

这条管线最重要的设计思想是：让可编程阶段负责灵活性，让固定功能阶段负责吞吐量。你可以写 Shader 控制顶点和像素怎么算，但三角形覆盖哪些像素、深度如何比较、颜色如何混合，这些高频规则通常交给硬件。

Draw Call 不是“画一个物体”的同义词，它是一次提交命令。一个 Draw Call 可以画一大批实例，也可能只画一个小 UI。

顶点到片元

管线前半段解决“在哪里”的问题。顶点先被变换到相机和屏幕空间，三角形再被硬件拆成片元。

顶点变换

模型文件里的顶点通常处在 模型空间。比如一个角色的鼻尖坐标可能是 (0, 1.7, 0.2)，这个坐标只描述它相对于角色自身原点的位置。要把它画到屏幕上，需要经历一串空间变换：

顶点着色器最经典的工作就是矩阵乘法：

1

float4 clipPos = mul(UNITY_MATRIX_MVP, float4(positionOS, 1.0));

这里的 MVP 可以拆成：

  • M / Model Matrix：模型到世界。角色站在哪里、朝哪个方向、缩放多少。
  • V / View Matrix：世界到相机。把相机看成新坐标系的原点。
  • P / Projection Matrix：相机到裁剪空间。透视投影让远处更小。

顶点阶段不会把模型“填满”。它只处理离散的点，并把每个顶点附带的数据传下去，例如 UV、法线、切线、顶点色、骨骼权重。真正把三角形铺成像素候选点的是下一步。

裁剪装配

GPU 收到顶点着色器输出后，会把索引缓冲中的三个顶点装配成三角形。接下来会发生几件事：

  • 背面剔除：如果一个三角形背对相机，且材质开启了 Cull Back，它会被丢弃。
  • 视锥裁剪：超出相机裁剪空间的部分会被切掉。
  • 透视除法：把 clipPos.xyzw 变成 NDC = clipPos.xyz / clipPos.w。
  • Viewport 映射：把 [-1, 1] 范围的 NDC 映射到屏幕像素坐标。

为什么三角形是实时渲染的基本单位？因为三角形永远共面，插值简单，硬件实现稳定。四边形、多边形最终也会被拆成三角形。

光栅化

光栅化是整条管线里最关键的固定功能阶段。它回答一个朴素问题：这个三角形覆盖了屏幕上的哪些采样点？

被覆盖的采样点会生成 Fragment。Fragment 还不是最终像素，它只是一个“像素候选”。它带着插值后的属性，比如 UV、法线、世界坐标、颜色。

这里有两个常见误区：

  • 片元不是像素：一个像素可能有多个采样点，尤其开启 MSAA 时；一个片元也可能被深度测试丢弃。
  • GPU 不是真的画线填色：它是在规则网格上做覆盖测试、插值和并行计算。

插值也不是简单的屏幕空间线性平均。透视投影会改变远近关系，如果直接线性插 UV，远处纹理会明显扭曲。因此硬件会做 perspective-correct interpolation，用裁剪空间的 w 修正顶点属性，保证纹理和法线随透视关系正确变化。

为什么要透视校正

投影之后，屏幕空间里的线性距离已经不等于三维空间里的线性距离。近处的同样一段距离会占更多像素，远处会占更少像素。

如果 UV 直接在屏幕空间线性插值，纹理会像贴在橡皮布上一样被错误拉伸。硬件会利用每个顶点的 1/w 做校正，让插值重新符合透视关系。

下面这张 Unity 复现实验图故意破坏了 UV 的透视校正。可以看到条纹没有按照地面远近关系均匀收缩，而是被拉成弯曲的带状形变。

片元到缓冲

管线后半段解决“是什么颜色、能不能留下”的问题。片元先算出候选颜色，再经过深度、模板和混合规则，最后写进渲染目标。

片元着色

Fragment Shader，也常叫 Pixel Shader，负责计算片元颜色。一个典型 PBR 材质大致会做这些事：

  • 用 UV 采样 Albedo、Normal、Metallic、Roughness、AO 等纹理。
  • 把法线从切线空间变换到世界空间。
  • 读取光源、阴影、环境贴图、反射探针。
  • 计算漫反射、镜面反射、菲涅尔、微表面分布。
  • 输出颜色、透明度，或者写入 G-Buffer。

在 Forward 渲染里，片元着色器通常直接输出最终颜色。在 Deferred 渲染里，它可能先输出 G-Buffer，例如 BaseColor、Normal、Roughness、Depth，再由后续 Lighting Pass 统一计算光照。

片元阶段通常是性能热点，因为屏幕像素数量巨大。1920x1080 约 207 万像素，4K 约 829 万像素。如果有大量半透明、阴影、多光源、后处理和过度绘制，同一个屏幕位置可能被计算很多次。

输出合并

片元着色器算出颜色后，不能立刻写进屏幕。它要经过输出合并阶段，常见规则包括：

  • Depth Test：比较当前片元深度和深度缓冲，决定谁离相机更近。
  • Depth Write：当前片元通过测试后，是否更新深度缓冲。
  • Stencil Test：用模板值做区域遮罩，例如描边、传送门、UI 裁剪。
  • Blending：把源颜色和目标颜色混合，例如透明、粒子、加法光效。
  • Color Mask：控制是否写入 RGB/A 的某些通道。

这也是为什么不透明物体通常先画、透明物体后画：

  • 不透明物体可以写深度，后面的远处片元会被 Early-Z 提前丢弃。
  • 透明物体需要混合，经常不能简单写深度，而且通常要从远到近排序。

这里还有一个很值钱的优化叫 Early-Z：如果片元在深度测试中注定失败，GPU 可以在片元着色器执行前就把它丢掉，省下昂贵的纹理采样和光照计算。但它有前提，Shader 不能随意改写深度，不能大量使用 discard 破坏硬件对深度的提前判断。

透明物体、discard、手写深度都会让 Early-Z 更难发挥。片元瓶颈严重时，先看这些地方，比盲目删贴图更可靠。

帧缓冲

GPU 最终写入的不是“显示器”，而是显存中的缓冲区。常见缓冲包括：

  • Color Buffer：颜色，可能是 SDR 的 RGBA8，也可能是 HDR 的 RGBA16F。
  • Depth Buffer：深度，常见 24-bit 或 32-bit。
  • Stencil Buffer：模板，常和深度共用格式。
  • G-Buffer：延迟渲染中保存法线、材质参数等中间数据。
  • Motion Vector：给 TAA、Motion Blur、DLSS/FSR/XeSS 等使用。
  • Intermediate Render Targets：Bloom、SSAO、SSR、后处理链的中间图。

所以“一帧”在引擎内部往往不是一张图，而是一组图。最终能被显示器拿去扫出的，通常是交换链里的一个后缓冲。

后处理的本质是“拿上一张图当纹理，再画一张全屏图”。Bloom、色调映射、景深、运动模糊、抗锯齿、颜色分级，基本都属于屏幕空间处理。

显示输出

GPU 画完以后，图像仍然只是显存里的数据。Present、交换链和显示扫描，负责把这块数据变成屏幕上的光。

交换链

应用程序通常不会直接写前台正在显示的缓冲，而是写 Back Buffer。写完后调用 Present，把后缓冲交给显示系统。常见模式：

  • 双缓冲：一个前缓冲显示，一个后缓冲绘制。Present 时交换。
  • 三缓冲：多一个缓冲，减少 CPU/GPU 因等待显示刷新而卡住的概率，但可能增加延迟。
  • VSync：等待垂直刷新边界再 Present，避免画面撕裂。
  • VRR：可变刷新率，显示器配合 GPU 改变刷新间隔，降低撕裂和卡顿。

画面撕裂的原因很简单：显示器正在从上到下扫描旧帧时，前缓冲中途被换成了新帧，于是屏幕上半部分和下半部分来自不同帧。

同步策略

VSync 的关键是只允许在垂直空白期交换缓冲。60 Hz 显示器每 16.67 ms 给你一次交换窗口，如果 GPU 在 16.67 ms 内画完，就能稳定 60 FPS；如果它用了 18 ms，错过这一班车，就要等下一个 VBlank，体感帧率会直接掉到 30 FPS 附近。这不是 GPU 只能画 30 帧，而是同步策略把不完整的中间结果按住了。

三缓冲的思路是多准备一个后缓冲，让 GPU 在等待显示扫描时仍然有地方继续画。它能减少因为 VBlank 等待造成的空泡，但可能增加输入到显示之间的排队延迟。G-Sync、FreeSync 这类 VRR 技术则反过来让显示器跟着 GPU 的完成节奏刷新，目标是同时降低撕裂、卡顿和等待延迟。

为什么会掉到 30 FPS

以 60 Hz 为例，显示器每 16.67 ms 扫描一帧。如果 GPU 在 17 ms 才完成，虽然只慢了 0.33 ms，但已经错过当前 VBlank，只能等下一次交换窗口。

所以传统双缓冲 + VSync 下，帧率经常不是平滑地从 60 掉到 55，而是直接落到 30 附近。三缓冲和 VRR 都是在处理这个同步颗粒度问题。

屏幕发光

到显示侧时，图像已经是一块二维颜色数组。每个像素通常包含红、绿、蓝三个颜色分量。显示控制器按固定时序读取像素数据，通过接口传给显示面板，例如 HDMI、DisplayPort、eDP、MIPI 等。

屏幕面板再把每个像素拆成更细的 RGB 子像素：

  • LCD：背光常亮或分区控制，液晶层调节每个子像素透光量，彩色滤光片形成 RGB。
  • OLED：每个子像素自发光，通过电流强弱控制亮度。
  • Mini LED / Micro LED：控制方式不同，但目标仍是把 RGB 数字值转成光强。

这里还牵涉颜色空间：

  • GPU Shader 常在线性空间做光照计算，因为真实光照能量近似线性叠加。
  • 普通 SDR 图像常以 sRGB 编码保存和显示，它不是线性亮度。
  • HDR 渲染会先在高动态范围缓冲中累积亮度，再经过 Tone Mapping 压到显示设备可表达范围。
  • Gamma、色域、亮度、面板校准都会影响你最终看到的颜色。

所以“Shader 输出 float4(1, 0, 0, 1)”并不等于你肉眼看到一个绝对物理意义的红色。它只是渲染系统中的一个颜色编码，后面还要经过色彩管理、显示传输函数和面板响应。

Shader 里算的是编码后的颜色值，不是人眼最终感知。线性空间、sRGB、HDR、Tone Mapping 混在一起时，颜色问题会变得很脏。

最小例子

用一个红色三角形，就能把前面的硬件、管线、缓冲和显示串起来。复杂游戏只是把这条路径重复很多次，并插入更多 Pass。

十步流程

假设我们要画一个红色三角形：

  1. CPU 创建顶点缓冲：三个顶点，位置分别是 A/B/C。
  2. CPU 绑定材质：使用一个简单 Shader，输出红色。
  3. CPU 提交 Draw Call：告诉 GPU 画 1 个三角形。
  4. Vertex Shader 跑 3 次：把 3 个顶点从模型空间变到裁剪空间。
  5. 固定功能阶段装配三角形：判断朝向、裁剪、映射到屏幕。
  6. Rasterizer 扫描三角形覆盖的像素采样点，生成许多片元。
  7. Fragment Shader 跑很多次：每个片元输出红色。
  8. Depth Test 决定红色片元是否比已有内容更近。
  9. Blend/ROP 把通过测试的颜色写入 Color Buffer。
  10. Present 后显示器读取 Color Buffer，RGB 子像素发光，人眼看到红色三角形。

这就是 GPU 画一帧的骨架。复杂游戏只是把这件事重复成千上万次，并插入阴影、后处理、粒子、UI、反射、全局光照等额外 Pass。

现代管线

上面讲的是经典光栅化管线。真实项目里，它正在被几类新路径扩展。

Mesh 管线

Mesh Shader 把传统的 Input Assembler、Vertex Shader、Geometry Shader 一部分工作改造成更接近 Compute 的可编程阶段。它适合做 GPU-Driven Culling、LOD 选择和小批量图元生成，典型收益是减少 CPU 逐物体提交 Draw Call 的压力。

光追路径

Ray Tracing 则走另一条路：不是从三角形覆盖像素，而是从像素发射光线，在 BVH 加速结构里找相交三角形。RT Core 这类专用硬件主要加速 BVH 遍历和射线三角形求交，常用于反射、阴影、全局光照等难用传统光栅化准确表达的效果。

Compute

Compute Shader 完全绕开固定图形管线，把 GPU 当通用并行处理器使用。现代引擎用它做剔除、粒子、后处理、Hi-Z、SSR、TAA、GPU Skinning，甚至整套可见性系统。

VRS

Variable Rate Shading 的思路更直接：不是每个屏幕区域都需要 1x1 精度着色。运动模糊区域、远景、周边视觉区域可以用 2x2 或更粗的着色率换吞吐，再把预算留给玩家真正注意的地方。

性能定位

优化不是删效果比赛。先判断瓶颈在 CPU、顶点、片元、带宽还是同步，再动对应的结构和资源。

找瓶颈

当一帧超过预算，例如 60 FPS 下超过 16.67 ms，就会掉帧。常见瓶颈可以按管线定位：

瓶颈典型原因直觉判断
CPU BoundDraw Call 太多、脚本逻辑重、剔除差、主线程等待降低分辨率 FPS 不明显提升
Vertex Bound顶点过多、蒙皮复杂、曲面细分重降低模型 LOD 有明显收益
Fragment Bound分辨率高、透明多、后处理重、复杂光照降低分辨率 FPS 明显提升
Texture Bound纹理采样多、缓存命中差、各向异性高减少采样或改善纹理布局有效
Bandwidth BoundMRT 多、HDR 格式大、频繁读写 Render Target降低 RT 格式或 Pass 数有效
SynchronizationCPU/GPU 等待、读回、屏障过多Profiler 中出现长等待或 pipeline bubble

性能优化不能靠玄学。正确路径是先用工具定位瓶颈：Unity Profiler、Frame Debugger、RenderDoc、Xcode GPU Capture、Nsight、Radeon GPU Profiler。要知道时间花在哪个 Pass、哪个 Draw、哪个 Shader，而不是凭感觉删效果。

没有 Profile 的优化基本是赌博。先证明瓶颈，再动代码；否则很容易把画质删没了，帧时间却不动。

对症优化

几个工程上立刻能用的判断：

  • Draw Call 多：优先看 SRP Batcher、GPU Instancing、Indirect Draw、材质排序和静态合批，但不要为了合批牺牲剔除粒度。
  • Fragment 压力大：先降分辨率验证，再看透明 Overdraw、后处理链、复杂 PBR 分支、阴影采样次数。
  • Bandwidth 爆：检查 Render Target 格式、MRT 数量、MSAA、HDR 中间纹理、频繁 Blit 和没有必要的全屏 Pass。
  • 同步等待：警惕 CPU 读回 GPU、过细的资源屏障、Present 阻塞、Frame Debugger 中看到的 pipeline bubble。

易混概念

Draw Call 不是三角形数量

Draw Call 是一次提交命令。一个 Draw Call 可以画很多三角形，也可能只画一个小 UI。过多 Draw Call 会增加 CPU 和驱动开销，但合批也不是免费的，会影响剔除、材质切换和内存布局。

Fragment Shader 次数不等于屏幕像素数

过度绘制、透明、MSAA、阴影、后处理都会让着色次数超过最终像素数。一个 4K 屏幕是 829 万像素，但一帧中实际执行的片元计算可能远超这个数量。

Framebuffer 不等于显示器

Framebuffer 是显存中的渲染目标。显示器只是最终读取其中一个可展示缓冲。很多中间 Framebuffer 永远不会直接显示。

Shader 算的是颜色，不是物理世界本身

PBR 只是用一套近似模型模拟光与材质。实时渲染一直在精度、速度、稳定性之间折中。

总结

GPU 画一帧不是某个单点技术，而是跨 CPU、GPU、显存、交换链和显示器的协作。理解它，后面的 Shader 优化和渲染管线分析才有坐标系。

一句话

GPU 画一帧，就是 CPU 把场景拆成渲染命令，GPU 用并行管线把顶点变成三角形、把三角形光栅化成片元、用 Shader 算出颜色、通过深度和混合写入帧缓冲，最后显示系统把这张二维颜色数组扫描到屏幕，驱动 RGB 子像素发光。

真正理解这条链路以后，Shader、渲染管线、后处理、抗锯齿、性能优化就不再是散点知识。它们都只是这条流水线上的某个阀门、缓存或计算阶段。
