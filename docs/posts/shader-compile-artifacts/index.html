Shader 编译不是“把一段 HLSL 直接变成显卡指令”。在 Unity 工程里，它更像一条分层流水线：ShaderLab 描述渲染状态，HLSL 描述可编程阶段，Unity 生成变体，平台编译器产出 DXBC、DXIL、SPIR-V 或 Metal 相关产物，最后驱动再把这些产物降到具体 GPU 能执行的机器码。

总览

前面几篇已经把 Shader 的入口、管线和写法拆开了。现在要回答一个更底层的问题：一个 .shader 文件进入 Unity 后，最终到底变成了什么？

答案不能只说“编译成二进制”。因为这条链路至少有四个边界：

  • 源码边界：开发者编写的是 ShaderLab、HLSL、Surface Shader、Shader Graph 生成代码，或 Compute Shader。
  • Unity 边界：Unity 会导入资产、展开 include、处理宏、生成 Surface/Graph 代码，并按 Pass、Stage、Keyword、平台图形 API 生成变体。
  • 平台边界：Direct3D、Vulkan、Metal 需要不同格式的 shader 表示，例如 DXBC、DXIL、SPIR-V、MSL 或 MetalLib。
  • 驱动边界：显卡驱动继续把平台产物编译成具体 GPU 的内部机器码，包含寄存器分配、指令调度、硬件特性适配。

一句话

DXBC、DXIL、SPIR-V、MSL、MetalLib 都不是统一意义上的最终显卡指令。 它们更接近 API 或平台运行时能消费的中间产物。真正贴着硬件 ISA 的代码通常在驱动内部生成，受 GPU 型号、驱动版本、编译策略、Pipeline State 和运行时缓存影响。

这件事很重要。在 Unity 里查看 compiled shader，或在 RenderDoc 里查看 SPIR-V/DXBC 反汇编，能判断宏是否展开、变体是否选对、采样次数是否偏多、控制流是否异常。但无法据此断言某张显卡最终用了哪条硬件指令、占用多少真实寄存器、如何排布波前。

看到 DXBC / SPIR-V 不等于看到最终硬件 ISA。它们是有价值的调试证据，但不是驱动私有编译器之后的终点。

源码层

Unity 的 Shader 源码并不是纯 HLSL 文件。常规 .shader 文件同时包含 ShaderLab 壳层和 HLSL 程序段；.compute 文件更接近单独的 Compute Shader；Shader Graph 和 Surface Shader 还会先生成一层代码。

Shader "Demo/CompilePath"

    Properties
    {
        _BaseMap ("Base Map", 2D) = "white" {}
        _Tint ("Tint", Color) = (1, 1, 1, 1)
    }

    SubShader
    {
        Tags { "RenderType"="Opaque" "Queue"="Geometry" }

        Pass
        {
            ZWrite On
            Cull Back

            HLSLPROGRAM
            #pragma vertex vert
            #pragma fragment frag
            #pragma shader_feature_local_fragment _ALPHATEST_ON
            ENDHLSL
        }
    }
}

这里的 Tags、Pass、ZWrite、Cull、Blend 不会原样变成 HLSL 指令。它们更像 Unity 渲染系统和底层图形 API 的状态描述：决定这个 Pass 什么时候被选中、如何绑定管线状态、是否写深度、如何剔除三角形、如何混合颜色。

struct Attributes
{
    float4 positionOS : POSITION;
    float2 uv : TEXCOORD0;
};

struct Varyings
{
    float4 positionCS : SV_POSITION;
    float2 uv : TEXCOORD0;
};

TEXTURE2D(_BaseMap);
SAMPLER(sampler_BaseMap);
float4 _Tint;

Varyings vert(Attributes input)
{
    Varyings output;
    output.positionCS = UnityObjectToClipPos(input.positionOS.xyz);
    output.uv = input.uv;
    return output;
}

float4 frag(Varyings input) : SV_Target
{
    float4 baseColor = SAMPLE_TEXTURE2D(_BaseMap, sampler_BaseMap, input.uv);
    return baseColor * _Tint;
}

这段代码看起来简单，但编译时会吸入大量上下文：Unity include 文件、管线宏、目标 shader model、图形 API、启用的关键字、当前 Pass 的 LightMode、SRP Batcher 的常量缓冲布局等。

Fx 名字

这里用 “Fx” 指代历史上的 effect / shader source 习惯，而不是说 Unity 直接把 .fx 文件作为主路径。Direct3D 时代常见的 FX 框架把 shader 函数、technique、pass、状态组合放在一个 effect 文件里。Unity 的 ShaderLab 在工程角色上有相似之处：它把“GPU 程序”和“渲染状态组织”放在同一个资产里。

差异也要讲清楚：Unity 不是 D3DX Effect 的薄封装。ShaderLab 是 Unity 自己的资产格式；Surface Shader、Shader Graph、URP/HDRP include、变体裁剪和平台交叉编译都由 Unity 的导入与构建系统参与。

Unity 层

Unity 的入口处理由 ShaderImporter 承担。它不会把文本一股脑交给显卡驱动，而是先做资产级处理。这个阶段解决的是“这个 shader 在 Unity 项目中定义了哪些可编译程序”。

导入处理

导入阶段常见工作包括：

  • 解析 ShaderLab：识别 SubShader、Pass、Tags、Properties、Fallback 和 UsePass。
  • 生成高层代码：Surface Shader 生成实际顶点/片元和光照相关 Pass；Shader Graph 生成 HLSL。
  • 建立依赖：记录 include 文件、管线库、宏和资产依赖，便于缓存和增量编译。
  • 收集编译入口：每个 Pass 内的 vertex、fragment、geometry、hull、domain、compute kernel 都可能形成独立 shader program。
  • 读取关键字声明：multi_compile、shader_feature、局部关键字、阶段后缀等会影响变体集合。

Unity 官方文档把这件事说得很直接：编辑器工作时不会预先编译所有变体，因为每个平台、每个图形 API、每个关键字组合都编完会明显变慢。导入时通常只做必要处理，真正需要某个变体时才查缓存或触发编译。

预处理

预处理是源码变成“可送入后端编译器”的关键一步。它会处理：

  • #include：把 UnityCG、Core.hlsl、Lighting.hlsl 等库纳入当前编译单元。
  • #define 与 #if：根据平台、管线、关键字、shader model 裁掉不走的代码。
  • 宏展开：例如采样宏、纹理声明宏、矩阵转换宏会变成后端更能理解的形式。
  • 平台符号：SHADER_API_D3D11、SHADER_API_VULKAN、SHADER_API_METAL 这类宏会让同一份 HLSL 在不同平台走不同分支。

这一步的危险在于：开发者以为自己写了一份 shader，实际上编译器看到的是多份条件化后的源码。很多“某个平台才报错”的问题，本质都是预处理后的代码已经不是肉眼看到的那一版。

变体

变体是 Unity Shader 编译成本的核心放大器。一个 Pass 不只是编译一次，而是按关键字组合、图形 API、目标平台和阶段入口扩展。

flowchart TD
  A[ShaderLab / HLSL 源码] --> B[ShaderImporter]
  B --> C[Surface / Graph 代码生成]
  C --> D[预处理 include / macro]
  D --> E{Pass}
  E --> F{Stage}
  F --> G{Keywords}
  G --> H{Graphics API}
  H --> I[平台 shader 产物]
  I --> J[驱动编译与缓存]
  J --> K[GPU 执行机器码]

#pragma shader_feature 常用于材质实际需要才保留的功能，#pragma multi_compile 常用于运行时必须保证存在的组合。后者很容易制造指数级膨胀。Unity 在构建时会做 stripping，但裁错变体会导致运行时找不到精确版本，只能回退或出现渲染错误。

变体不是“if 的另一种写法”。它是在构建期生成多份专用 shader program，用编译体积和加载成本换运行时分支成本。

平台产物

经过 Unity 处理后，shader 会进入平台编译或翻译路径。不同 Unity 版本、不同渲染后端、不同平台工具链会有差异，因此工程判断不能停留在记忆某个编译器名字上。更稳妥的理解是：Unity 要把统一的 ShaderLab/HLSL 资产，转成目标图形 API 能创建 pipeline 的 shader 表示。

Direct3D

在 Direct3D 11 时代，常见平台产物是 DXBC。它是 DirectX shader bytecode，能被 D3D runtime 和驱动继续消费。在工具里看到的 dcl_input、sample、mad、ret 等形式，通常属于这类中间层反汇编视图。

Direct3D 12 和较新的 HLSL 工具链里，DXIL 变得更常见。DXIL 基于 LLVM 风格的中间表示，携带验证、优化和更现代的 shader model 支持。它仍然不是某张 NVIDIA、AMD 或 Intel GPU 的最终 ISA，而是 D3D12 运行时和驱动之间的 shader 表示。

简化理解：

名称常见语境边界
HLSL开发者源码或 Unity 生成源码人写或引擎生成
DXBCD3D11 常见 shader bytecodeAPI 可消费中间码
DXILD3D12 / 新 shader model 常见中间表示API 可验证中间码
GPU ISA厂商硬件指令驱动私有最终层

若要优化 D3D shader，DXBC/DXIL 很有价值：它能反映纹理采样次数、分支结构、常量缓冲访问、是否出现预期宏。但它不能直接反映某个 wave 在 RDNA、Ada、Arc 上最终如何调度。

Vulkan

Vulkan 的核心 shader 产物是 SPIR-V。它是 Khronos 定义的二进制中间表示，模块化、可验证，也更适合跨语言前端。HLSL 可以通过编译器前端转成 SPIR-V，GLSL 也可以转成 SPIR-V。

SPIR-V 的重要特点是：它已经越过了源码层，但还没有越过驱动最终编译层。Vulkan 驱动拿到 SPIR-V 后，仍要结合 pipeline state、descriptor layout、subgroup 能力、目标 GPU 做进一步优化和机器码生成。

RenderDoc 在 Vulkan 项目里经常能看到 SPIR-V、GLSL 反编译视图，甚至在带调试信息时做源码级调试。这些工具适合定位：

  • 当前 draw 绑定的是哪个 shader module。
  • specialization constant 或 descriptor 是否符合预期。
  • 变体分支是否已经被编译期裁掉。
  • 纹理、UBO、SSBO 的绑定点是否和管线布局一致。

但如果目标是“查看最终硬件指令”，SPIR-V 仍然只到 API 边界。某些厂商工具可能展示更靠近 ISA 的视图，但那已经进入厂商工具链，不应把 RenderDoc 里看到的 SPIR-V 当成通用硬件事实。

Metal

Metal 的路径和 D3D/Vulkan 不完全一样。Apple 平台使用 Metal Shading Language，也就是 MSL。Unity 会把项目 shader 转成 Metal 可接受的表示，再由 Apple 的工具链生成可用于运行时创建 pipeline 的产物。常见最终打包形态会涉及 MetalLib 或 pipeline 相关缓存。

这里容易误判：看到 MSL 源码并不意味着看到硬件代码；看到 MetalLib 也不意味着它就是 Apple GPU 的最终 ISA。Metal 驱动、系统版本、设备型号、pipeline state 仍会影响最终编译和缓存。

Xcode 的 Metal Debugger / GPU Frame Capture 可以辅助定位 Metal 层问题：

  • 当前 render pass / compute pass 绑定了哪些 pipeline。
  • shader 函数、资源绑定、texture、buffer 是否符合预期。
  • 某些设备上是否出现编译、验证或资源访问问题。
  • GPU workload 的瓶颈更像 fragment、texture、bandwidth 还是同步。

但它不是 Unity ShaderImporter 的替代品。Unity 侧的关键字、Pass 选择、变体裁剪错误，仍要回到 Unity compiled shader、Frame Debugger 或构建日志里定位。

扩展性说明：为什么不要绑定某个后端产物

如果项目要长期跨 D3D、Vulkan、Metal，Shader 架构不应该把业务逻辑绑定在某一种平台产物上。更稳的做法是把可复用数学、采样约定、空间变换、材质参数布局放在 Unity HLSL include 层，再用少量平台宏处理确实不同的边界。

例如纹理采样、深度范围、坐标翻转、半精度策略、绑定模型，应该集中在项目 shader 库里，而不是散落到每个材质文件。这样当 Unity 后端、目标平台或图形 API 迁移时，改动面会小得多。

驱动层

真正执行 shader 的不是 DXBC、DXIL、SPIR-V 或 MSL 文本，而是 GPU 能理解的机器指令。这个转换通常发生在驱动、系统图形栈或厂商编译器内部。

最终 ISA

驱动最终编译会处理很多源码层看不到的问题：

  • 寄存器分配：临时变量、插值输入、常量、纹理坐标如何落到寄存器。
  • 指令选择：高层 normalize、pow、矩阵乘法、采样操作如何拆成硬件指令或特殊函数单元调用。
  • 指令调度：如何隐藏纹理延迟、ALU 延迟、内存访问延迟。
  • 分支处理：静态分支、动态分支、分支发散如何映射到 wave / warp 执行。
  • 资源绑定：descriptor、argument buffer、constant buffer、sampler 状态如何进入硬件绑定表。
  • Pipeline 关联：blend、depth、raster state、render target 格式可能影响最终 pipeline 编译。

同一份 SPIR-V 在两张 GPU 上生成不同 ISA 是正常的。同一份 Metal shader 在不同 Apple SoC、不同系统版本上的最终优化也可能不同。驱动更新后性能变化，也常来自这层编译器行为。

缓存时机

Unity 在编辑器和构建期有自己的 shader cache；图形 API 和驱动也可能有 pipeline cache 或 shader cache。它们不是同一个缓存。

可以粗略拆成三层：

层级缓存内容常见用途
Unity Library/ShaderCacheUnity 编译过的平台变体结果加快编辑器和构建期迭代
API / Pipeline cachePipeline 或 shader module 相关编译结果降低运行时创建 pipeline 成本
驱动缓存厂商私有机器码或优化结果降低后续运行或同类 pipeline 编译成本

因此，“项目已经 build 完”不等于所有设备上都没有 shader 编译成本。移动端、主机、PC 驱动都可能在首次遇到某些 pipeline 时做额外工作。项目越依赖运行时动态组合材质、关键字和 render state，越需要关注预热和缓存策略。

编译卡顿很多时候不是 HLSL 语法问题，而是变体数量、Pipeline State 组合和驱动首次编译叠在一起。只关注源码行数，容易定位错环节。

查看方法

调试 Shader 编译产物时，第一步不是打开工具，而是明确问题发生在哪一层。下面这些方法都能用，但边界不同。

Unity 里看

在 Unity 里，最直接的方法是查看 shader 资产的 Inspector：

  • 对手写 .shader，使用 Inspector 里的 compiled shader / compile and show code 入口，选择目标平台或图形 API，查看 Unity 生成后的平台代码。
  • 对 Surface Shader，先看 generated code，确认 Unity 自动生成了哪些 Pass、哪些输入输出、哪些光照函数。
  • 对 Shader Graph，使用生成代码视图确认 Graph 实际生成的 HLSL，而不是只看节点连线。
  • 对构建问题，查看 Editor.log 或构建日志中的 shader compilation / stripping 信息，确认哪些变体被编译或裁掉。

这里能验证的问题很具体：

  • #ifdef SHADER_API_METAL 是否走了预期分支。
  • _ALPHATEST_ON 这类关键字是否真的生成了对应变体。
  • Surface Shader 是否额外生成 ShadowCaster、Meta、Deferred 等 Pass。
  • 某个 include 宏展开后是否引入了平台不支持的语法。

不要把 Unity compiled shader 当作硬件性能报告。它适合查“Unity 到平台产物”的路径，不适合直接推断最终占用、wave 数量或具体 ISA。

RenderDoc 看

RenderDoc 更适合在帧捕获里回答“当前 draw 实际绑定了什么”：

  • 捕获一帧，选中目标 draw call。
  • 查看 Pipeline State，定位 vertex / pixel 或 compute shader。
  • 在 shader viewer 中查看反汇编、中间表示或可用的源码映射。
  • 对照资源绑定，确认 texture、buffer、sampler、constant buffer 是否对应材质参数。

RenderDoc 的强项是把 shader 和 draw call、资源、render target 放在同一个上下文里。它能抓到“Unity 里看着没问题，实际帧里绑定错了”的问题。例如材质关键字没开、SRP pass 没走、RenderTexture 格式不对、常量缓冲值异常。

它的边界同样明确：RenderDoc 常展示 API 层 shader 表示和反编译视图，不保证提供最终硬件 ISA。若要查看 AMD、NVIDIA、Intel 的更底层信息，应使用对应厂商工具，并且结论要限定到具体 GPU 与驱动版本。

Xcode 看

Apple 平台上，Xcode GPU Frame Capture / Metal Debugger 是定位 Metal 层问题的主工具。它适合检查：

  • Unity 生成的 Metal pipeline 是否成功创建。
  • 当前 render command encoder 绑定了哪些资源。
  • texture、buffer、sampler、argument buffer 的绑定是否符合预期。
  • shader 调试和性能视图是否指向 fragment、texture、bandwidth 或 tile memory 相关瓶颈。

不要假设所有 Metal 工具视图在所有设备和系统版本上都暴露同样深度。Xcode 能定位到 Metal API 和 GPU 调试层，但 Unity 变体生成、ShaderLab Pass 选择、SRP 内部调度仍要结合 Unity 侧工具看。

可验证清单

实际排查时，我会按这个顺序走：

  1. 在 Unity Frame Debugger 里确认 draw call 是否存在、Pass 是否正确、材质关键字是否符合预期。
  2. 在 shader Inspector 查看 compiled shader 或 generated code，确认预处理和变体结果。
  3. 用 RenderDoc 或 Xcode 捕获真实帧，确认运行时绑定的 shader、资源和 render target。
  4. 若问题是平台差异，分别检查 D3D、Vulkan、Metal 目标下的预处理宏和平台产物。
  5. 若问题是性能，先用 GPU profiler 定位瓶颈，再决定是否进入厂商 ISA 工具；不要从一段反汇编直接猜结论。

这个顺序的核心是先验证“是不是同一个 shader、同一个变体、同一个 Pass”，再谈优化。很多项目把时间浪费在分析错误的变体上，这是常见却代价昂贵的失误。

工程结论

Shader 编译产物的真正价值，是帮助建立边界意识。

四层边界

可以把整条链路压缩成四句话：

  • ShaderLab / HLSL 是作者意图：它描述材质、Pass、阶段函数和可维护的源码结构。
  • Unity 生成与变体是工程现实：它决定实际有多少 Pass、多少阶段、多少关键字组合、多少平台副本。
  • DXBC / DXIL / SPIR-V / Metal 产物是 API 契约：它们能被图形 API、运行时和驱动消费，也能被工具检查。
  • 驱动机器码是硬件事实：它通常不完全暴露，且依赖厂商、设备、驱动和 pipeline state。

写法建议

为了让这条链路长期可控，项目里应该遵守几条硬规则：

  • 少用无边界的 multi_compile，能用 shader_feature 就不要制造永久变体。
  • 平台差异集中封装在项目 include，不要散落在每个材质 shader。
  • 关键 shader 至少检查一次目标平台 compiled code，尤其是移动端和 Metal。
  • 性能结论必须绑定工具证据：Frame Debugger 证明 Pass，RenderDoc/Xcode 证明运行时绑定，Profiler 证明瓶颈。
  • 不要把某个平台的中间码细节写成跨平台业务假设。

这一节并不要求记住每个后端的所有文件格式。真正要记住的是：Shader 编译是一条分层流水线，每一层都有自己的输入、输出和调试工具。把层次分清，很多“Unity shader 难以解释的问题”就会收敛为可验证的工程问题。
